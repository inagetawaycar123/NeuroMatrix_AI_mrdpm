# NeuroMatrix AI 卒中智能体升级方案

## 一、升级后的总体定位

NeuroMatrix AI 下一阶段应从原来的“卒中辅助诊断系统 / 固定流程 Demo”，升级为：

> 面向基层医院急性卒中首诊场景的 DAG 编排式、可追踪、可校验、可迭代的卒中专病智能体系统。

系统通过 **DAG 编排** 和 **Skill 调用**，将 NCCT 分诊、血管堵塞识别、类 CTP 生成、卒中自动分析、指南检索、报告生成、人机复核和反馈迭代组织成一条可解释、可验证、可追溯的临床辅助决策链路。

核心关键词：

- 受约束 DAG 编排
- Skill Registry
- 多 Agent 协作
- 分级知识库
- 证据追溯
- Agent Cockpit
- 人机协同
- 自我迭代

---

## 二、升级后的系统总架构

建议将系统升级为 **六层架构**：

```text
① 前端交互层
病例任务窗口 / DAG执行图 / Agent Cockpit / 影像查看 / 证据面板 / 报告编辑 / 医生反馈

② 服务与任务管理层
病例管理 / 文件上传 / Agent Run管理 / 任务状态管理 / API服务 / WebSocket实时推送

③ 智能体编排层
Triage Planner / DAG Orchestrator / Executor / RePlanner / Consistency Reviewer / Human Review Controller

④ Skills能力层
NCCT三分类 / 血管堵塞三分类 / MRDPM类CTP生成 / 卒中自动分析 / 指南检索 / 报告生成 / 问答 / 归档

⑤ 知识与模型层
分级知识库 / 混合RAG检索 / 小型卒中知识图谱 / 模型注册表 / 规则库 / 证据库

⑥ 数据与审计层
case_data / agent_run / node_run / skill_result / evidence_items / report_payload / feedback / logs / hard_case_pool
```

这一版架构的重点不是把模块堆在一起，而是让每个模块都能被智能体编排、被前端追踪、被医生复核、被系统记录。

# 三、核心架构设计一：DAG 编排

## 3.1 DAG 在系统中的作用

DAG 不是为了炫技，而是为了解决一个核心问题：

> 不同病例输入不同，不能所有病例都走同一条固定流程。

例如：

- 有的病例只有 NCCT；
- 有的病例有 NCCT + CTA/mCTA；
- 有的病例已经有 CTP；
- 有的病例疑似脑出血，不能继续走缺血性卒中的溶栓分析路径；
- 有的病例中间结果冲突，需要触发人工复核。

因此，系统需要从固定流程：

```text
上传影像 → MRDPM生成 → 自动分析 → 报告生成
```

升级为动态编排：

```text
病例输入 → Planner判断输入条件 → 动态生成DAG → 调用对应Skills → 校验 → 汇总 → 医生确认
```

---

## 3.2 推荐的 DAG 主流程

```text
病例创建
  ↓
模态识别 Skill
  ↓
NCCT三分类 Skill
  ↓
是否疑似脑出血？
  ├── 是 → 出血预警路径 → 医生复核 → 终止AIS分析路径
  └── 否 → 继续AIS路径
          ↓
      是否存在CTA/mCTA？
          ├── 是 → 血管堵塞三分类 Skill
          └── 否 → 建议补充CTA/转诊评估
          ↓
      是否已有真实CTP？
          ├── 有 → 跳过MRDPM，直接卒中分析
          └── 无 → MRDPM生成类CTP
          ↓
      梗死核心/缺血半暗带分析
          ↓
      指南/知识库检索
          ↓
      一致性校验
          ↓
      结构化报告生成
          ↓
      医生确认/修改
          ↓
      归档与反馈学习
```

系统由此从“固定工作流”升级为“受临床规则约束的动态工作流”。

---

## 3.3 DAG 节点数据结构设计

每个节点建议保存以下信息：

```json
{
  "node_id": "node_003",
  "node_name": "NCCT三分类",
  "skill_name": "skill_ncct_triage_classification",
  "status": "success",
  "input": {},
  "output": {},
  "confidence": 0.87,
  "risk_level": "medium_high",
  "need_human_review": true,
  "evidence_items": [],
  "start_time": "",
  "end_time": "",
  "error_message": ""
}
```

这样前端可以展示：

- 当前执行到哪一步；
- 哪一步成功；
- 哪一步失败；
- 哪一步需要医生确认；
- 每一步输出了什么；
- 每一步用了什么证据。

# 四、核心架构设计二：Skill Registry

## 4.1 Skill Registry 的作用

需要将原来的功能模块，改造成智能体可以调用的标准能力。

原来的方式可能是后端写死：

```python
run_mrdpm()
run_analysis()
generate_report()
```

升级后应注册为标准 Skills：

```text
skill_detect_modality
skill_ncct_triage_classification
skill_vessel_occlusion_classification
skill_generate_ctp_maps
skill_stroke_analysis
skill_retrieve_guidelines
skill_consistency_check
skill_generate_structured_report
skill_request_human_review
skill_archive_case
```

Planner 不直接写死流程，而是根据病例情况选择要调用哪些 Skill。

---

## 4.2 建议保留的 Skills

| Skill | 作用 | 优先级 |
|---|---|---|
| 模态识别 Skill | 判断上传的是 NCCT、CTA、mCTA、CTP | P0 |
| 图像质控 Skill | 判断图像是否缺失、格式是否正确 | P0 |
| NCCT 三分类 Skill | 脑出血 / 脑缺血 / 正常初筛 | P0 |
| 血管堵塞三分类 Skill | 大血管堵塞 / 中血管堵塞 / 正常 | P0 |
| MRDPM Skill | 无 CTP 条件下生成类 CTP 图 | P0 |
| 卒中分析 Skill | 梗死核心、半暗带、mismatch 等分析 | P0 |
| 指南检索 Skill | 从分级知识库中检索相关依据 | P1 |
| 一致性校验 Skill | 检查模型、报告、规则是否冲突 | P1 |
| 报告生成 Skill | 生成结构化报告 | P0 |
| 医生复核 Skill | 请求医生确认、修改或驳回 | P0 |
| 归档 Skill | 保存病例、报告、日志、版本 | P0 |
| 反馈学习 Skill | 将医生修改沉淀为迭代数据 | P1 |

# 五、核心架构设计三：多 Agent 系统

## 5.1 多 Agent 的实施原则

不建议一开始就做十几个复杂 Agent。

更合理的方式是：

> 先做“多 Agent 角色设计 + 单执行器实现”，后续再拆成真正多 Agent。

也就是说，当前版本可以在代码上先由一个 Orchestrator 调度，但在架构上划分清楚 Agent 角色。

---

## 5.2 推荐 Agent 设计

### 1. Triage Planner Agent

负责判断病例该走哪条路径。

输入：

```text
病例信息 + 影像模态 + 临床症状 + 时间窗
```

输出：

```text
DAG任务图
```

---

### 2. Imaging Agent

负责：

- 影像模态识别；
- 图像质控；
- NCCT 三分类；
- 血管堵塞三分类。

---

### 3. Perfusion Agent

负责：

- MRDPM 类 CTP 生成；
- 输出 CBF、CBV、Tmax 等类灌注图。

---

### 4. Stroke Analysis Agent

负责：

- 梗死核心分析；
- 缺血半暗带分析；
- mismatch ratio 计算；
- 关键量化指标提取。

---

### 5. Guideline Agent

负责：

- 指南检索；
- 医院 SOP 检索；
- 临床知识补充；
- 证据来源标注。

---

### 6. Consistency Reviewer Agent

负责检查不同模块结果是否矛盾。

例如：

```text
NCCT提示脑出血，但报告仍然建议溶栓 → 强制拦截

血管堵塞提示正常，但灌注图显示大面积异常 → 请求医生复核

模型置信度低但报告语气过于肯定 → 降低结论强度
```

---

### 7. Report Agent

负责生成结构化报告。

注意：

> Report Agent 只能负责表达和结构化组织，不能直接替代医学判断。

---

### 8. Human Review Agent

负责：

- 医生确认；
- 医生修改；
- 医生驳回；
- 反馈记录。

---

### 9. Learning Agent

负责将医生反馈进入迭代流程：

```text
错误归因 → 难例库 → 待审核知识池 → 版本更新
```

医疗场景中的 Learning Agent 不能自动改规则，必须经过人工审核。

---

# 六、核心架构设计四：分级知识库 + 混合检索

评委提出“知识库文件如何分级、权重如何处理”，说明系统不能只做普通 RAG。

下一步建议将知识库改造成 **分级证据库**。

---

## 6.1 知识库分级设计

| 等级 | 来源 | 权重 | 用途 |
|---|---|---:|---|
| S级 | 国家/国际卒中指南、诊疗规范 | 最高 | 决策依据、适应证、禁忌证 |
| A级 | 合作医院本地 SOP、转诊流程 | 很高 | 本地化流程适配 |
| B级 | 专家共识、教材、综述 | 中高 | 医学解释、背景知识 |
| C级 | 脱敏病例、历史报告、医生修改记录 | 中等 | 案例参考、表达学习 |
| D级 | 模型生成内容、普通问答记录 | 低 | 仅作参考，不作为医学依据 |

---

## 6.2 检索得分设计

不要只看语义相似度，可以设计为：

```text
最终得分 =
语义相似度
+ BM25关键词匹配
+ 权威等级权重
+ 时效性权重
+ 场景匹配权重
- 冲突惩罚
```

例如医生问：

> 这个病人是否适合进一步取栓评估？

系统应该优先检索：

1. 卒中取栓指南；
2. 医院本地转诊流程；
3. 当前病例影像分析结果；
4. 相关病例经验。

---

# 七、核心架构设计五：卒中知识图谱

知识图谱不建议一开始做大。  
建议先做一个“小而精”的 **卒中首诊知识图谱**。

---

## 7.1 节点设计

```text
疾病节点：
脑出血 / 脑缺血 / 正常 / AIS / LVO / MeVO

影像节点：
NCCT / CTA / mCTA / CTP / 类CTP

指标节点：
CBF / CBV / Tmax / core volume / penumbra volume / mismatch ratio

治疗节点：
溶栓 / 取栓 / 转诊 / 保守治疗 / 进一步检查

规则节点：
时间窗 / 禁忌证 / 适应证 / 风险提示

证据节点：
指南条目 / 医院SOP / 论文 / 医生修改记录 / 典型病例
```

---

## 7.2 关系设计

```text
脑出血 → 禁忌 → 静脉溶栓

大血管闭塞 → 提示 → 取栓评估

中血管闭塞 → 建议 → 专科复核

Tmax延迟 → 支持 → 缺血半暗带判断

mismatch升高 → 支持 → 存在可挽救组织

NCCT正常 → 不能排除 → 早期AIS

指南条目 → 支撑 → 临床建议

医生修改 → 反馈 → 难例库
```

---

## 7.3 知识图谱自学习机制

“自学习”应定义为：

> 医生反馈驱动、人工审核确认、版本可追溯的安全更新机制。

流程：

```text
医生修改报告
  ↓
系统记录修改点
  ↓
错误归因
  ↓
进入待审核知识池 / 难例库
  ↓
医生或管理员审核
  ↓
更新规则库 / 知识图谱 / 训练集
  ↓
生成新版本
```

这可以回答评委关于“系统如何自我迭代”的问题。

# 八、新增 NCCT 三分类与血管堵塞三分类的处理方式

这两个模型不要作为孤立功能加入，而要作为 **首诊入口分诊 Skills**。

---

## 8.1 NCCT 三分类的定位

NCCT 三分类任务：

```text
脑出血 / 脑缺血 / 正常
```

它是整个系统的第一道安全分流。

DAG 位置：

```text
病例输入 → 模态识别 → NCCT三分类 → 路径选择
```

输出逻辑：

```text
脑出血：
高风险预警，终止AIS溶栓分析路径，请求医生复核。

脑缺血：
进入AIS深度分析路径，继续血管堵塞判断和类CTP生成。

正常：
不能直接排除AIS，提示结合症状、时间窗和CTA/mCTA继续评估。
```

特别注意：

> NCCT 正常不能写成“无卒中”。

应写成：

> NCCT 未见明确异常，但早期急性缺血性卒中不能完全排除，建议结合 CTA/mCTA、临床症状和发病时间继续评估。

---

## 8.2 血管堵塞三分类的定位

血管堵塞三分类任务：

```text
大血管堵塞 / 中血管堵塞 / 正常
```

它是血管层面的风险评估。

DAG 位置：

```text
NCCT三分类后 → CTA/mCTA输入 → 血管堵塞三分类
```

输出逻辑：

```text
大血管堵塞：
高优先级预警，触发取栓评估/上转建议。

中血管堵塞：
中高风险提示，建议专科复核和灌注分析。

正常：
结合类CTP、临床症状和时间窗继续判断。
```

---

## 8.3 两个新增模型与 MRDPM 的关系

不能将它们和 MRDPM 讲成重复关系。

正确关系是：

```text
NCCT三分类：
判断是否出血、是否疑似缺血，是安全分诊入口。

血管堵塞三分类：
判断血管层面是否存在闭塞风险。

MRDPM：
判断组织灌注层面是否存在低灌注、缺血核心和半暗带。

卒中分析：
将灌注图和影像结果转化为量化指标和临床结论。
```

升级后的逻辑是：

```text
安全分诊 → 血管判断 → 灌注评估 → 组织分析 → 指南校验 → 报告生成
```

# 九、前端交互设计

前端的核心目标是：

> 让评委和医生看见智能体如何规划、执行、校验、解释和请求确认。

不要只做：

```text
上传 → 等待 → 输出报告
```

这太像普通工具。

应该做成：

```text
病例任务窗口 + DAG执行图 + 节点卡片 + 证据面板 + 报告编辑 + 人机复核 + 反馈闭环
```

---

## 9.1 页面一：病例列表页

功能：

- 新建病例；
- 查看历史病例；
- 查看当前状态；
- 查看风险等级；
- 查看是否需要医生确认。

示例字段：

```text
病例ID
患者脱敏编号
输入模态
当前任务状态
风险等级
医生复核状态
创建时间
```

状态标签：

```text
待分析 / 分析中 / 待复核 / 已归档 / 分析失败
```

---

## 9.2 页面二：病例创建与上传页

上传内容：

- NCCT；
- CTA/mCTA；
- CTP，可选；
- 基本病史；
- 发病时间；
- 症状描述；
- NIHSS，可选。

上传后系统自动执行：

```text
模态识别 → 图像质控 → 创建 Agent Run
```

前端提示：

```text
已识别输入模态：NCCT + mCTA
建议执行路径：AIS首诊辅助分析路径
```

---

## 9.3 页面三：Agent Cockpit 总览页

这是最重要的页面。

建议分成四个区域：

```text
左侧：病例基本信息 + 输入模态 + 风险状态
中间：DAG执行图
右侧：当前结论 + 证据面板 + 风险提示
底部：运行日志 / 时间线
```

中间 DAG 显示：

```text
模态识别
  ↓
NCCT三分类
  ↓
血管堵塞三分类
  ↓
MRDPM类CTP生成
  ↓
卒中分析
  ↓
指南检索
  ↓
一致性校验
  ↓
报告生成
  ↓
医生确认
```

每个节点显示状态：

```text
已完成 / 运行中 / 等待中 / 失败 / 需要人工确认
```

---

## 9.4 页面四：节点详情卡片

点击 DAG 中任意节点，右侧弹出节点卡片。

卡片内容：

```text
节点名称
调用 Skill
输入数据
输出结果
置信度
风险等级
证据来源
耗时
是否需要人工复核
```

例如 NCCT 三分类节点：

```text
结果：疑似脑缺血
置信度：0.87
风险等级：中高
下一步：继续AIS深度分析
提示：NCCT早期缺血识别存在不确定性，请结合CTA/mCTA继续评估
```

---

## 9.5 页面五：影像查看与结果叠加页

支持：

- 原始 NCCT；
- CTA/mCTA；
- 类 CTP 图；
- CBF；
- CBV；
- Tmax；
- 梗死核心；
- 缺血半暗带；
- 叠加显示。

关键是不要只展示图，要展示“图和结论的关系”。

例如：

```text
结论：存在缺血半暗带
证据：Tmax延迟区域与CBF/CBV不匹配区域对应
```

---

## 9.6 页面六：证据追溯面板

每一个关键结论都要能展开证据。

示例：

```text
关键结论：
疑似急性缺血性卒中，伴大血管闭塞风险。

证据链：
1. NCCT三分类：脑缺血，置信度0.87
2. 血管堵塞三分类：疑似大血管闭塞，置信度0.91
3. MRDPM类CTP：Tmax延迟区域明显
4. 卒中分析：存在缺血半暗带
5. 指南检索：大血管闭塞且时间窗内，建议进一步取栓评估
```

这样评委能明显感受到：

> 系统不是黑盒输出，而是证据链输出。

---

## 9.7 页面七：结构化报告编辑页

报告不要只是一段自然语言。  
建议分成：

```text
基本信息
输入影像
模型分析结果
量化指标
关键结论
风险提示
```

# 十四、下一版答辩系统叙事

升级之后，可以这样介绍系统：

> 过去我们的系统更像一个固定流程的卒中辅助诊断工具；现在我们将它升级为 DAG 编排式卒中专病智能体。  
> 系统首先通过 NCCT 三分类完成安全分诊，再通过血管堵塞三分类判断血管风险；随后根据是否存在 CTP 动态选择真实 CTP 分析或 MRDPM 类 CTP 生成，再完成梗死核心和缺血半暗带分析。  
> 在此基础上，系统调用分级知识库和指南检索 Agent 进行证据补充，通过一致性校验 Agent 检查冲突，最终生成可解释、可追溯、可由医生确认的结构化报告。  
> 所有执行步骤都会在 Agent Cockpit 中展示，医生的修改和反馈会进入难例库、待审核知识池和模型迭代池，从而形成安全可控的持续优化闭环。
