# NeuroMatrix AI（卒中先锋智能影像诊断平台）智能体架构与技术方案说明

## 1. 文档概述

### 1.1 项目背景

NeuroMatrix AI 是一个面向急性缺血性脑卒中场景的智能影像诊断与辅助决策平台。项目围绕临床卒中救治的核心问题展开：如何基于患者上传的 [`NCCT`](../README.md)、[`mCTA`](../README.md) 及可选 [`CTP`](../README.md) 数据，快速完成灌注参数生成、核心梗死与半暗带量化分析、报告生成、问题驱动结论输出以及证据追溯展示，从而提升临床评估效率与结论透明度。

该项目并非单纯追求“生成一份报告”，而是强调：

1. 结论必须建立在结构化影像量化结果之上；
2. 关键判断必须经过多层校验与冲突收敛；
3. 最终结果必须具备可解释性、可追溯性与可审计性；
4. 系统设计必须能够适应后续模型替换、模态扩展和审核流程升级。

### 1.2 建设目标

本项目的建设目标可以归纳为四个层面：

| 目标层面 | 目标说明 |
|---|---|
| 临床效率 | 缩短从影像上传到量化分析、报告生成与问题回答的链路时间 |
| 结论可信 | 通过 ICV、EKV、Consensus 等机制降低单模型直接生成结论带来的幻觉风险 |
| 工程可落地 | 在现有 Flask + React + 推理模型能力上迭代，不推翻既有流程，支持真实部署 |
| 架构可演进 | 支持新增模型、新增模态、新增智能体与新增审核环节的平滑接入 |

### 1.3 平台定位

NeuroMatrix AI 的平台定位不是“单一大模型医学问答系统”，也不是“孤立的影像推理脚本集合”，而是一个面向卒中场景的 **多模型、多工具、多智能体协同的影像分析与结论生成平台**。

其定位具有以下特点：

1. **以影像量化结果为核心真值基础**：核心梗死体积、半暗带体积、不匹配比值、受累侧别等量化结果构成后续报告与结论的基础输入；
2. **以智能体编排为主线**：路径判定、工具执行、校验、收敛、汇总均由 Agent Loop 驱动；
3. **以问题驱动为输出导向**：不仅输出标准报告，还针对用户问题形成面向临床使用的定向结论；
4. **以证据链为可信保障**：所有关键结论需要映射到证据项或明确标记不可用原因。

### 1.4 适用场景

本平台主要适用于以下场景：

1. 急性缺血性脑卒中患者的初步影像分析；
2. 基于 NCCT + mCTA 的灌注参数补全与卒中定量评估；
3. 临床医生希望围绕具体问题获取结构化建议，如“是否存在可挽救脑组织”“是否建议进一步血管内治疗”；
4. 需要向院内评审、技术答辩或验收场景说明系统如何保证结论可信与可追溯。

## 2. 总体架构设计

### 2.1 总体分层

从工程实现看，系统可划分为五个层次：

```text
┌──────────────────────────────────────────────────────┐
│ 展示与交互层                                         │
│ React/Vite 报告页、问题驱动结论、证据追溯、AI 问答     │
├──────────────────────────────────────────────────────┤
│ 服务与编排层                                         │
│ Flask API、上传流程、Agent Run、任务调度、状态查询     │
├──────────────────────────────────────────────────────┤
│ 智能体协同层                                         │
│ Planner / Loop Controller / Context / Reporter      │
│ ICV / EKV / Consensus / Summary Assembler           │
├──────────────────────────────────────────────────────┤
│ 模型与工具层                                         │
│ MRDPM、Palette、Stroke Analysis、Baichuan、MedGemma  │
├──────────────────────────────────────────────────────┤
│ 数据与审计层                                         │
│ Supabase / PostgreSQL、report_payload、events、logs   │
└──────────────────────────────────────────────────────┘
```

这一分层并非形式化拆分，而是直接对应当前代码结构：

| 分层 | 当前实现位置 | 主要职责 |
|---|---|---|
| 展示层 | [`frontend/src/components/StructuredReport.tsx`](../frontend/src/components/StructuredReport.tsx)、[`frontend/src/components/MedicalAIChat.tsx`](../frontend/src/components/MedicalAIChat.tsx) | 报告展示、问题驱动结论、证据追溯、AI 问答交互 |
| 服务层 | [`backend/app.py`](../backend/app.py) | API 入口、上传编排、Agent 运行管理、报告生成接口 |
| 智能体层 | [`backend/agent/`](../backend/agent/) | Loop 执行、上下文维护、规划与结果整合 |
| 模型层 | [`mrdpm/`](../mrdpm/)、[`palette/`](../palette/)、[`backend/stroke_analysis.py`](../backend/stroke_analysis.py)、[`backend/medgemma_report.py`](../backend/medgemma_report.py) | 灌注生成、卒中量化分析、报告与视觉理解 |
| 数据层 | [`sql/`](../sql/)、Supabase 接口、运行态内存结构、`report_payload` | 患者数据、影像元数据、结果对象与审计轨迹 |

### 2.2 为什么采用分层架构

采用上述分层的核心原因在于，该项目同时具备“影像推理系统”“AI 报告系统”“多智能体编排系统”三种属性，如果不进行边界分层，极易出现以下问题：

1. 推理逻辑、API 逻辑、页面渲染逻辑混杂，导致维护成本过高；
2. 报告生成与证据校验强耦合，后续难以替换大模型或增加验证阶段；
3. 新增模态或工具时需要修改多处主流程，难以扩展；
4. 缺乏统一的运行态与上下文管理，智能体协作无法收敛。

当前分层架构的价值在于：

- 展示层专注“如何解释结果”；
- 服务层专注“如何暴露能力”；
- 智能体层专注“如何调度与收敛”；
- 模型层专注“如何产出分析结果”；
- 数据层专注“如何存储、回放、审计”。

这使得系统具备较好的可维护性和可替换性。

### 2.3 总体业务链路

平台的主链路可抽象为：

```text
影像上传
  -> 模态识别与路径判定
  -> CTP 灌注图生成（如需要）
  -> 脑卒中量化分析
  -> 报告生成
  -> 内部一致性校验（ICV）
  -> 外部知识验证（EKV）
  -> 共识裁决（Consensus）
  -> Summary Assembler 汇总
  -> 问题驱动结论生成
  -> 证据追溯组装
  -> 前端可视化展示
```

其中，报告生成并不是最终输出终点，真正面向临床使用的终态是：

1. 有结构化最终报告；
2. 有围绕用户问题的定向回答；
3. 有每条关键结论的证据映射；
4. 有冲突、不确定性和下一步建议的清晰标识。

## 3. 智能体架构设计

### 3.1 智能体划分原则

本项目的智能体划分并不是为了“形式上多智能体”，而是基于真实业务链路中不同阶段的目标函数不同：

| 划分原则 | 说明 |
|---|---|
| 单一职责 | 每个智能体聚焦一个阶段目标，降低逻辑混杂 |
| 可验证 | 每一阶段输出都应成为下一阶段的明确输入 |
| 可替换 | 某一智能体逻辑升级不应破坏整体链路 |
| 可审计 | 每个节点都能记录输入、输出、状态与失败原因 |
| 可收敛 | 多个智能体的结论冲突时，必须存在统一裁决机制 |

这也是为什么当前系统不是让一个大模型直接“读完所有数据后生成最终结论”，而是采用“规划—执行—校验—汇总”的多阶段机制。

### 3.2 当前智能体与核心组件

基于当前实现，智能体相关核心模块主要分布在 [`backend/agent/`](../backend/agent/) 下：

| 模块 | 实现文件 | 职责 |
|---|---|---|
| Planner | [`backend/agent/planner.py`](../backend/agent/planner.py) | 根据路径与运行上下文生成工具序列与计划帧 |
| Loop Controller | [`backend/agent/loop_controller.py`](../backend/agent/loop_controller.py) | 负责整个 Agent Loop 生命周期控制、重试、暂停、收敛 |
| Context Manager | [`backend/agent/context_manager.py`](../backend/agent/context_manager.py) | 维护运行上下文、事实状态、计划帧与重规划标记 |
| Tool Registry | [`backend/agent/tool_registry.py`](../backend/agent/tool_registry.py) | 维护工具集合与契约边界 |
| Executor | [`backend/agent/executor.py`](../backend/agent/executor.py) | 执行具体工具，并产出统一 Observation |
| Reporter | [`backend/agent/reporter.py`](../backend/agent/reporter.py) | 将最终执行结果整理为可对外消费的结果对象 |

从业务角色上看，当前系统可映射出以下智能体：

| 业务角色 | 对应实现 | 核心职责 |
|---|---|---|
| 主分析智能体 | Planner + Loop Controller | 路径判定、工具调度、链路推进 |
| 影像工具智能体 | `generate_ctp_maps`、`run_stroke_analysis` 等工具 | 完成模型推理与结构化分析 |
| 内部一致性校验智能体 | `icv` | 检查量化结果、路径、报告之间是否一致 |
| 外部知识验证智能体 | `ekv` | 将关键 claim 与外部知识/指南证据进行映射 |
| 共识裁决智能体 | `consensus_lite` | 面对冲突结果进行统一收敛 |
| 报告生成智能体 | `generate_medgemma_report` | 生成基础报告与结构化载荷 |
| 结果整合智能体 | [`build_summary_artifacts()`](../backend/summary_assembler.py:801) | 生成问题驱动结论、traceability、evidence map、final report |

### 3.3 智能体之间的关系

智能体之间并非并列独立运行，而是呈现明显的上下游依赖关系：

```text
Triage Planner Agent
  -> 决定本次病例属于哪条模态路径
  -> 生成可执行工具链

Clinical Tool Agent
  -> 生成 CTP 参数图 / 执行卒中分析 / 生成基础报告

Verification Agents
  -> ICV：验证内部一致性
  -> EKV：验证外部知识支撑
  -> Consensus：对冲突结果进行收敛

Summary / Conclusion Agent
  -> 汇总最终报告
  -> 生成问题驱动结论
  -> 组装证据追溯与 answer ledger
```

这套关系设计的关键价值在于：

1. **将“生成结果”与“验证结果”解耦**；
2. **将“结论表达”与“证据构建”解耦**；
3. **允许上游模型不变的前提下，增强下游校验能力**；
4. **允许未来新增新的审核/裁决智能体，而不破坏主分析链路**。

### 3.4 多智能体协作流程

结合 [`AgentLoopController.run()`](../backend/agent/loop_controller.py:49) 的执行逻辑，当前系统协作流程可以概括为以下时序：

#### 阶段 1：任务初始化

1. 后端创建 `run`，写入 `planner_input`；
2. 运行态中记录 `patient_id`、`file_id`、`available_modalities`、`question` 等基础输入；
3. Planner 读取病例上下文，生成初始计划帧与工具序列。

#### 阶段 2：主分析执行

1. 依据路径策略决定是否执行 [`generate_ctp_maps`](../backend/app.py)；
2. 执行 [`run_stroke_analysis`](../backend/app.py)，产出核心梗死、半暗带、不匹配比等结构化指标；
3. 执行 [`generate_medgemma_report`](../backend/app.py)，生成报告基础载荷。

#### 阶段 3：多层校验

1. `icv` 对内部结果进行一致性审查；
2. `ekv` 将关键结论映射到外部知识与文献证据；
3. `consensus_lite` 对 ICV/EKV 冲突进行轻量裁决，形成收敛决策。

#### 阶段 4：结果收敛与发布

1. [`summary_assembler.py`](../backend/summary_assembler.py:801) 基于 `icv/ekv/consensus/report_payload` 组装：
   - `final_report`
   - `evidence_items`
   - `evidence_map`
   - `traceability`
   - `question_answer`
   - `answer_evidence_ledger`
2. 前端读取统一的 `report_payload`，展示结构化报告、问题驱动结论与证据追溯。

### 3.5 冲突处理与结果收敛机制

多智能体系统最大的风险不是“没有结果”，而是“多个结果互相矛盾”。因此当前架构显式设置了三类收敛机制：

#### 3.5.1 非阻断校验机制

在 [`AgentLoopController`](../backend/agent/loop_controller.py:136) 中，`icv`、`ekv`、`consensus_lite` 被定义为非阻断型校验工具。其设计逻辑是：

1. 校验失败并不等于主链路必须中断；
2. 在临床真实场景中，系统应尽量给出可用但带不确定性标识的结果；
3. 校验失败本身也是重要的风险信号，应进入最终报告，而不是直接丢弃。

#### 3.5.2 Consensus 收敛机制

当 ICV 和 EKV 对关键结论存在冲突时，通过 `consensus_lite` 给出统一裁决结果，例如：

- `accept`
- `review_required`
- `escalate`

该设计避免前端同时收到多个互相冲突的 verdict，提升结果一致性与展示稳定性。

#### 3.5.3 高风险暂停机制

在 Agent Loop 中保留 `pause_on_high_risk` 能力。当存在高风险未支持 claim 或 Consensus 指向升级处理时，系统可以进入人工复核暂停态，而非继续自动推进。这使得系统具备向“人机协同审核流程”过渡的基础能力。

### 3.6 为什么多智能体优于单一大模型直接输出

相较于“把所有数据扔给一个大模型，直接生成结论”的做法，当前多智能体架构具有明显优势：

| 对比维度 | 单一大模型直出 | 当前多智能体架构 |
|---|---|---|
| 过程透明度 | 低 | 高，可分阶段查看 |
| 幻觉控制 | 弱 | 强，存在多层校验 |
| 可审计性 | 差 | 强，支持事件回放 |
| 可替换性 | 弱，模型强耦合 | 强，工具与校验可独立替换 |
| 可扩展性 | 新需求易导致 prompt 膨胀 | 新能力可作为工具/智能体接入 |

因此，多智能体并非为了增加系统复杂度，而是为了把复杂度组织起来，使其可管理、可复用、可验证。

## 4. 技术方案设计

### 4.1 前端技术方案与展示逻辑

前端采用 React + TypeScript + Vite，核心目标不是单纯展示文本，而是将后端生成的结构化结果以临床可读方式呈现。

#### 4.1.1 前端结构特点

前端主要位于 [`frontend/src/`](../frontend/src/)：

| 组件 | 作用 |
|---|---|
| [`StructuredReport.tsx`](../frontend/src/components/StructuredReport.tsx) | 报告主容器，渲染患者信息、影像发现、问题驱动结论、证据追溯 |
| [`ImageFindingsModule.tsx`](../frontend/src/components/ImageFindingsModule.tsx) | 影像发现与量化结果展示 |
| [`PatientInfoModule.tsx`](../frontend/src/components/PatientInfoModule.tsx) | 患者基本信息展示 |
| [`MedicalAIChat.tsx`](../frontend/src/components/MedicalAIChat.tsx) | 临床 AI 问答组件 |
| [`report.css`](../frontend/src/styles/report.css) | 报告页整体样式与模块卡片样式 |

#### 4.1.2 展示层设计原则

前端展示遵循以下原则：

1. **结构化优先于富文本堆叠**：`report_payload` 中的对象优先以模块化卡片展示，而非纯 markdown 文本；
2. **结论与证据并列展示**：问题驱动结论与证据追溯作为独立模块并列出现，避免“看见结论却找不到来源”；
3. **状态可视化**：风险等级、置信度、覆盖率、裁决结果均可被可视化表达；
4. **面向临床阅读流**：用户问题、AI 回答、关键要点、下一步建议、不确定性按照临床决策阅读顺序排列。

### 4.2 后端服务编排与接口设计

后端以 Flask 为主，核心编排逻辑集中在 [`backend/app.py`](../backend/app.py)。

#### 4.2.1 核心接口形态

当前服务并未将每一类能力拆成独立微服务，而是在单体 Flask 中按“能力边界”封装接口，典型包括：

| 接口类型 | 说明 |
|---|---|
| 上传相关接口 | 负责影像上传、任务创建、上传进度查询 |
| Agent 运行接口 | `/api/agent/runs*`，负责创建、查询、重试、拉取结果 |
| 报告相关接口 | 负责生成结构化报告与 viewer 展示 |
| 校验上下文接口 | 为 validation、cockpit 等页面聚合运行结果 |
| AI 问答接口 | 提供基于百川 M3 的流式/同步问答能力 |

#### 4.2.2 为什么采用“单体服务 + 内部工具编排”

对于当前项目阶段，采用单体后端而不是拆分多个服务，主要基于以下考虑：

1. 影像推理、数据库访问、报告组装、智能体运行高度耦合，拆分服务会显著增加集成成本；
2. 当前重点是打通临床主链路并强化可信化能力，而不是过早进行分布式拆分；
3. 通过模块化代码组织（如 `backend/agent/`、`summary_assembler.py`）已经实现逻辑上的边界解耦；
4. 后续如需服务化拆分，可基于现有工具边界逐步演进。

### 4.3 医学影像处理与模型推理链路

#### 4.3.1 输入模态与路径判定

系统支持以下路径：

1. `NCCT-only`
2. `NCCT + single-phase-CTA`
3. `NCCT + mCTA`
4. `NCCT + mCTA + CTP`

路径判定遵循模态优先级与冻结规则，这一设计确保不同病例不会误入错误推理链路，减少无效计算和异常调用。

#### 4.3.2 CTP 生成链路

当病例为 `NCCT + mCTA` 且未上传真实 CTP 时，系统通过 MRDPM / Palette 生成灌注参数图：

```text
mCTA 输入
  -> generate_ctp_maps
  -> MRDPM / Palette 推理
  -> 输出 CBF / CBV / Tmax
  -> 写入结果目录与后续分析上下文
```

这一设计使平台在缺少真实 CTP 的场景下，仍具备生成灌注参数的能力，提升落地场景适用性。

#### 4.3.3 脑卒中量化分析链路

量化分析由 [`backend/stroke_analysis.py`](../backend/stroke_analysis.py) 等模块完成，核心输出包括：

1. `core_infarct_volume`
2. `penumbra_volume`
3. `mismatch_ratio`
4. `hemisphere`

这些结构化量化结果是后续报告生成、问题驱动结论与证据验证的基础真值输入。之所以将其单独结构化输出，而非在报告阶段现算现写，是为了保证：

- 上游分析结果可复用；
- 下游校验逻辑可以稳定消费；
- 报告与问题结论共享同一组基础数据，避免“双源不一致”。

### 4.4 大模型接入方式与结构化输出思路

#### 4.4.1 大模型的角色定位

项目中大模型并不负责“从原始影像直接做一切”，而是承担以下角色：

| 模型 | 角色 |
|---|---|
| 百川 M3 | 报告生成、问题驱动结论生成、临床问答 |
| MedGemma | 医学影像语义理解、报告生成辅助 |

这意味着大模型更多承担“语言组织、语义提升、面向临床表达”的职责，而不是替代底层定量推理。

#### 4.4.2 为什么采用“结构化输入 + LLM 表达增强”

如果直接让大模型基于原始文本自由生成医学结论，风险较高。当前设计采用：

```text
结构化量化数据 + 患者上下文 + 校验结果 + 问题语义
  -> Prompt 组装
  -> LLM 生成
  -> 结构化字段回写 / 前端展示
```

其价值在于：

1. 让 LLM 站在已有量化事实之上组织表达；
2. 降低无约束生成带来的幻觉概率；
3. 方便后续对 prompt、字段、格式进行配置化管理；
4. 支持问题导向的动态回答，而不是单一固定模板报告。

### 4.5 报告生成与结果落盘/回显

报告生成并不是独立终点，而是被纳入整个 `report_payload` 统一对象中。该对象既承担后端结果承接，又承担前端单一真值来源。

#### 4.5.1 `report_payload` 作为单一真值

从 [`summary_assembler.py`](../backend/summary_assembler.py:819) 可见，系统统一以 `report_payload` 为基础对象，逐步补齐：

1. `final_report`
2. `evidence_items`
3. `evidence_map`
4. `traceability`
5. `question_answer`
6. `answer_evidence_ledger`

这一设计非常关键，因为它避免了：

- 报告文本一份、校验结果一份、问题结论一份、前端展示一份之间的多源漂移；
- 前端需要对多个接口结果进行复杂合并；
- 不同模块更新时产生字段语义不一致。

#### 4.5.2 结果回显策略

前端报告页优先读取结构化 `report_payload`，当本地缓存或 API 返回发生变化时进行统一解析。这种回显策略提高了页面容错性与展示一致性。

## 5. “问题驱动结论”机制专项说明

### 5.1 为什么采用问题驱动方式

传统影像报告面向“完整描述”，但临床实际使用往往面向“具体问题”。例如：

- 是否存在可挽救脑组织？
- 是否支持进一步再通治疗？
- 当前证据是否足以支持血管内治疗？

如果系统只能生成标准报告，而不能围绕问题进行归纳与判断，则临床使用者仍需自行从长文本中抽取关键结论，认知负担较高。

因此，问题驱动结论模块的设计目标是：

1. 将量化分析与校验结果转化为面向问题的回答；
2. 让系统输出更贴近真实临床决策场景；
3. 保持回答与原始量化结果、校验链路的一致性。

### 5.2 当前实现位置与架构角色

问题驱动结论的核心生成逻辑位于 [`_build_question_answer()`](../backend/summary_assembler.py) 与 [`build_summary_artifacts()`](../backend/summary_assembler.py:801) 所在流程中。

其在整体架构中的位置如下：

```text
量化分析结果 + ICV/EKV/Consensus
  -> Summary Assembler
  -> 问题聚焦 claim 识别
  -> LLM 或规则增强回答生成
  -> question_answer 写入 report_payload
  -> 前端“问题驱动结论”模块展示
```

也就是说，问题驱动结论并不是独立游离在报告链路之外的附加功能，而是最终结果整合层的一部分。

### 5.3 如何把影像分析转化为问题结论

当前实现包含以下关键步骤：

1. **接收用户问题**：通过 Agent Run 的 `planner_input.question` 写入运行上下文；
2. **识别问题焦点**：根据问题关键词匹配关键 claim，例如 `significant_mismatch`、`treatment_window_notice`；
3. **提取相关结论**：从关键 findings 中筛选最相关的证据支持项；
4. **综合患者上下文**：补充年龄、性别、NIHSS、发病至入院时间等信息；
5. **调用 LLM 生成临床可读回答**：优先走百川 M3，失败则回退规则增强回答；
6. **输出结构化结果**：包括 `question`、`direct_answer`、`key_points`、`recommendations`、`uncertainties`、`confidence`。

### 5.4 对临床可读性与透明度的提升

问题驱动机制的价值，不在于“把报告再说一遍”，而在于：

1. 明确回答用户真正关心的问题；
2. 将底层量化值与临床建议建立直接联系；
3. 明确呈现支持点、不确定性和下一步建议；
4. 将回答与证据映射绑定，提升临床信任度。

这也是为什么前端中“问题驱动结论”被设计为独立于通用报告正文的模块，而不是埋在报告段落内部。

## 6. “证据追溯”机制专项说明

### 6.1 为什么必须引入证据追溯

在医疗 AI 场景中，仅给出一个结论是不够的。临床用户最关心的问题是：

- 这个结论来自哪里？
- 是模型猜测、规则推断，还是有明确证据支撑？
- 如果没有证据，系统有没有明确标记风险？

证据追溯机制的目标，就是建立“结论—证据”的明确映射，提升系统可解释性与可信度。

### 6.2 证据来源构成

当前项目中的证据来源主要包括：

| 证据来源 | 说明 |
|---|---|
| 量化分析结果 | 核心梗死体积、半暗带体积、不匹配比等结构化结果 |
| ICV 发现 | 内部一致性校验中的异常、警告与建议动作 |
| EKV claims / citations | 对关键 claim 的外部知识支撑、文献片段、文档页码 |
| Consensus 决策 | 对冲突结论的统一裁决结果 |
| 指南文档库 | `EKV_docs` 下卒中相关规范与共识文件 |

### 6.3 结论—证据映射建立方式

在 [`build_summary_artifacts()`](../backend/summary_assembler.py:801) 中，系统完成以下映射动作：

1. 基于 EKV 输出构建 `evidence_items`；
2. 为每个关键 claim 建立 `claim_to_evidence_ids`；
3. 生成 `evidence_map`，记录 `finding_id -> evidence_ids / unavailable_reason`；
4. 统计 `traceability.coverage`、`mapped_findings`、`unmapped_ids`、`high_risk_unmapped_count`；
5. 输出给前端用于“证据追溯”模块展示。

### 6.4 如何向用户展示证据支撑

前端“证据追溯”模块并不只是展示一组 UUID，而是以以下方式组织信息：

1. 风险等级与置信度摘要；
2. 追溯覆盖率与映射总数；
3. 逐条关键发现卡片；
4. 每条发现对应的 verdict、message、evidence 引用或 unavailable 原因。

这一设计说明，证据追溯并不是后台审计专属信息，而是前台可视化展示的重要组成部分。

### 6.5 如何降低幻觉风险与提升可信度

证据追溯机制在降低幻觉风险方面发挥了直接作用：

1. **要求关键结论有证据或有原因**：高风险结论不能“无来源悬空存在”；
2. **将不确定性显式暴露**：对于无证据支撑的项，不进行掩盖，而是输出 unavailable reason；
3. **把 LLM 结果锚定到结构化 claim**：避免语言层自由发挥脱离底层事实；
4. **支持人工审核与回放**：当结论存在争议时，可直接回溯 claim 与证据映射。

### 6.6 证据追溯在整体架构中的位置与价值

证据追溯并不是“汇总阶段附加的展示功能”，而是整套多智能体系统得以成立的重要前提。没有证据追溯，多智能体即使输出了结果，也无法证明其可信度。

因此，它在架构中承担三重作用：

1. **对内**：作为多智能体结论收敛的审计依据；
2. **对前端**：作为解释层内容直接展示给用户；
3. **对评审**：作为医疗 AI 合规与可信化能力的关键支撑。

## 7. 可维护性设计

### 7.1 模块解耦

当前工程在可维护性方面最重要的设计之一，是通过目录和模块边界进行解耦：

| 解耦维度 | 当前实现 |
|---|---|
| 前后端解耦 | React 前端位于 [`frontend/`](../frontend/)，Flask 后端位于 [`backend/`](../backend/) |
| 智能体与业务工具解耦 | Agent 编排逻辑在 [`backend/agent/`](../backend/agent/)，业务工具仍在 [`backend/app.py`](../backend/app.py) 中统一适配 |
| 汇总与报告解耦 | [`summary_assembler.py`](../backend/summary_assembler.py:801) 负责结果整合，不侵入底层推理逻辑 |
| 展示与数据对象解耦 | 前端围绕 `report_payload` 读取，不直接依赖后端内部中间状态 |

### 7.2 配置管理

系统通过 `.env` 和相关环境变量管理运行策略，例如：

- `AGENT_LOOP_V2_ENABLED`
- `AGENT_LOOP_V2_CANARY_RATIO`
- `AGENT_LOOP_V2_PLANNER_MODE`
- `AGENT_HUMAN_REVIEW_PAUSE`
- `BAICHUAN_API_KEY`

这种配置化设计的意义在于：

1. 运行策略可调整，而非写死在代码里；
2. 不同部署环境可采用不同灰度与审核策略；
3. 大模型接入与 Agent 行为可进行无代码切换。

### 7.3 接口边界与契约化

项目中大量使用结构化对象作为契约边界，例如：

- `RunState`
- `AgentEvent`
- `EvidenceItem`
- `FinalReport`
- `ErrorContract`

这使得：

1. 工具输出更容易被上游/下游消费；
2. 前后端协作时字段语义更稳定；
3. 后续新增能力时可以以“增量字段”形式演进，而不破坏旧协议。

### 7.4 日志、调试与回放

系统在运行态中保留了如下可调试信息：

1. `run_id` 级别的运行状态；
2. `events` 级别的执行轨迹；
3. `[SUMMARY]`、`[EVIDENCE]`、`[ANSWER]` 等关键日志；
4. `report_payload` 中的 `decision_trace`、`tool_metrics`、`traceability` 等字段。

这些日志与轨迹信息使得开发者在调试异常时，不必重新阅读整条链路的所有代码，而可以从 `run_id` 出发进行回放与定位。

### 7.5 文档与目录组织

项目已在 [`docs/`](../docs/) 下沉淀多个交付文档，包括：

- [`docs/W4_ICV交付.md`](../docs/W4_ICV交付.md)

## 8. 可拓展性设计

### 8.1 模态扩展

当前路径体系已经围绕 `available_modalities` 做了归一化和路径判定，这意味着后续新增模态时，可以通过以下方式接入：

1. 在模态识别阶段增加归一化映射；
2. 在路径决策中扩充合法路径；
3. 为新模态补充对应工具与分析逻辑；
4. 在汇总层定义新的关键 claim 与证据映射规则。

因此，模态扩展不是重写整条链路，而是在现有路径规则上扩容。

### 8.2 模型替换

项目的模型层并未与展示层直接耦合。只要新模型仍能输出兼容的结构化字段，例如：

- `core_infarct_volume`
- `penumbra_volume`
- `mismatch_ratio`
- `hemisphere`

就可以平滑替换现有推理模型，而不影响：

1. Agent 调度框架；
2. Summary Assembler；
3. 前端问题驱动结论与证据追溯模块。

### 8.3 智能体新增

当前架构天然支持新增智能体，例如：

1. 人工审核前置智能体；
2. 诊疗建议专用智能体；
3. 多指南比对智能体；
4. 结果解释与教学反馈智能体。

新增方式通常为：

1. 增加工具或校验节点；
2. 在 Planner 或 Loop 中纳入新步骤；
3. 将输出并入 `report_payload`；
4. 在前端增加对应展示模块。

### 8.4 审核流扩展

当前系统已经具备高风险暂停与人工复核的基础能力。未来若需要支持更完整的审核流，可扩展为：

```text
自动分析
  -> 智能体校验
  -> 高风险暂停
  -> 人工审核节点
  -> 审核意见回写
  -> 二次收敛
  -> 最终发布
```

这一方向与真实医疗 AI 落地场景高度契合。

### 8.5 前端展示扩展

由于前端围绕 `report_payload` 的结构化对象渲染，因此后续若新增：

- 新的证据分组；
- 新的临床建议模块；
- 新的风险评分卡片；
- 新的推理过程可视化组件；

均可以以独立组件方式接入，而不需要重写整个报告页。

## 9. 异常处理与系统稳健性设计

### 9.1 异常处理原则

本项目的异常处理不是简单“报错退出”，而是遵循以下原则：

1. 能回退则回退；
2. 能继续则继续；
3. 不能继续时必须给出错误契约；
4. 即便失败，也要保留足够上下文用于追溯。

### 9.2 典型异常处理策略

| 异常类型 | 处理方式 |
|---|---|
| 模态缺失 | 路径判定失败或走兼容路径 |
| CTP 生成失败 | 给出错误契约，必要时中断主链路 |
| ICV/EKV 失败 | 非阻断，进入软失败记录 |
| LLM 调用失败 | 回退到规则增强回答 |
| 高风险冲突 | 可触发人工复核暂停 |
| 前端字段缺失 | 按 unavailable 或默认值回退展示 |

### 9.3 为什么这种异常设计更适合医疗 AI 场景

医疗 AI 场景不适合“一旦有局部失败就完全无结果”，因为临床决策往往仍希望获得部分有效信息。但同时，也不能将失败悄悄吞掉。因此当前异常策略实现了“保留可用信息 + 明确风险边界”的平衡，更适合真实落地。

## 10. 技术方案合理性总结

### 10.1 为什么当前方案适合该项目

当前方案之所以适合 NeuroMatrix AI，原因在于它同时满足了该项目的三类需求：

1. **业务需求**：围绕卒中影像分析、报告生成与问题回答形成完整闭环；
2. **可信需求**：通过 ICV、EKV、Consensus 与证据追溯机制提高结论可信度；
3. **工程需求**：在已有代码基础上逐步演进，避免推倒重来。

### 10.2 相比单一大模型直接生成结论的优势

当前方案相较于单一大模型直出结论，主要优势体现在：

1. 有明确的结构化真值输入；
2. 有中间过程可验证、可回放；
3. 有结论与证据的映射关系；
4. 有冲突处理与人工审核扩展空间；
5. 更符合医疗 AI 对可信性与可解释性的要求。

### 10.3 面向真实医疗 AI 场景的工程价值

从工程角度看，该项目的价值不只是完成了一条卒中分析链路，更重要的是形成了一套可迁移的方法论：

1. 用结构化结果约束大模型生成；
2. 用多智能体机制组织复杂流程；
3. 用证据追溯机制支撑可信输出；
4. 用配置化与模块化设计保障系统可持续演进。

这使得 NeuroMatrix AI 不仅适合作为当前卒中项目的落地平台，也具备向更多医学影像场景扩展的工程基础。

## 11. 结论

NeuroMatrix AI 的架构设计并非围绕“模型越多越强”展开，而是围绕“如何让结果更可信、流程更清晰、系统更可落地”展开。通过将影像推理、智能体编排、问题驱动结论、证据追溯与前端解释层进行系统化整合，平台构建了一条从原始影像到临床可读结论的完整技术链路。

这一方案的核心优势在于：

1. 设计思路清晰，模块边界明确；
2. 校验与汇总机制完整，可信化能力突出；
3. 前后端与模型层解耦，维护与扩展成本可控；
4. 面向真实医疗 AI 落地场景，具备持续演进与工程交付价值。

对于技术评审、项目答辩与方案汇报场景而言，该项目不仅展示了具体功能实现，更体现了在高风险医疗 AI 场景中，如何通过多智能体协作与证据链机制，将“能生成结果”升级为“能解释结果、能校验结果、能交付结果”的系统化能力。
