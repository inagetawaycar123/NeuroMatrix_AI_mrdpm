# StrokeClaw 前端交互优化方案

## 一、目标定义

在比赛展示与评审场景中，评委首先感知到的不是系统底层是否采用了更复杂的 Agent 编排、Skill 抽象或审计机制，而是前端交互体验是否明显区别于传统“上传影像—等待结果—输出报告”的工具模式。因此，StrokeClaw 的前端优化目标，不是单纯美化页面，而是让评委与用户在第一眼就感受到：

**这不是一个一次性输出结果的卒中分析工具，而是一个会接手病例任务、先规划、再执行、途中汇报、关键处请求确认、最后形成闭环的专病智能体系统。**

前端优化的核心任务，是把后端的“编排、执行、校验、协同、闭环”翻译成用户可感知的交互体验。

---

## 二、评委在前端应当看到的关键变化

如果评委看到的仍然只是“上传影像 → 等待 → 报告生成”，那么项目在前端层面仍然表现为升级版工具，而不是 StrokeClaw。要让评委真正感知到 StrokeClaw 的成立，前端至少应体现以下七个变化。

### 1. 从“上传入口”变成“病例任务入口”
首页不再只是上传文件，而应首先展示一个“待处理卒中病例任务”对象，包括：

- 病例编号
- 患者基础信息
- 当前影像模态组合
- 当前状态
- 风险等级
- 启动按钮：**启动 StrokeClaw 分析流程**

这样，系统从一开始就不再像一个“函数调用界面”，而更像一个“病例任务工作台”。

### 2. 从“黑盒等待”变成“显式执行计划”
用户点击启动后，不应直接进入长时间 loading，而应首先展示 `Stroke Orchestration` 面板，明确说明本次病例将依次经历哪些步骤，例如：

1. 病例接收  
2. 影像质控  
3. 模态路由  
4. MRDPM 生成  
5. 卒中自动分析  
6. 证据与规则校验  
7. 结构化结论生成  
8. 报告草拟  
9. 医生复核  
10. 回写与归档  

这样评委会立即理解：系统不是“后台偷偷算完再吐结果”，而是在执行一条显式的任务计划。

### 3. 从“最终报告”变成“运行中的节点网络”
执行过程中，每个步骤都应以节点卡片形式呈现，例如：

- `Image_QC.validate()`
- `Modality_Router.route()`
- `MRDPM_Generate.run()`
- `Stroke_Analysis.segment()`
- `Evidence_Check.verify()`
- `Human_Review.await_action()`
- `EMR_Sync.writeback()`

评委看到的将不再是一页静态报告，而是一个多节点协同运行的病例处理过程。

### 4. 从“只给结论”变成“关键中间量可见”
前端必须主动展示关键中间结果，而不是只展示最后结论。建议展示至少包括：

- 序列识别与质控结果
- 模态路由结果
- MRDPM 生成状态
- core / penumbra / mismatch 等关键指标
- 风险提示与低置信提示
- 证据校验结果

这样，评委会感受到系统是在“一步一步走到这个结论”，而不是直接“给出一个答案”。

### 5. 从“指出问题”变成“发现问题后继续推进”
当前端发现风险或信息缺失时，不能只显示“请补充信息”，而应同时给出下一步建议，例如：

- 建议补录：LKW（最后正常时间）
- 建议补录：NIHSS
- 建议人工复核：指定切片与区域
- 建议进入：医生复核模式
- 建议生成：待补充版结构化报告

这能体现系统不是只会报错，而是会持续推进任务。

### 6. 从“最后点确认”变成“正式人机协同节点”
医生确认不应只是最后一个按钮，而应是流程中的正式节点，例如 `Human_Review.await_action()`。前端应清晰展示：

- 待确认项
- AI 建议
- 高风险项
- 证据支持
- 可选动作：采纳、修改、驳回、转交复核

这样评委会更容易感知到项目不是“替代医生”，而是“与医生协同”。

### 7. 从“输出结束”变成“病例闭环完成”
流程结束后，前端不应只显示“报告已生成”，而应显示：

- 病例处理完成
- 关键结论已确认
- 最终报告已归档
- 结果已回写 / 可导出 / 可进入随访

这会把项目从“生成工具”提升为“闭环系统”。

## 三、StrokeClaw 前端的推荐信息架构

建议将前端交互结构设计为：

**1 个病例首页 + 4 个核心页面 + 1 个病例级问答能力**

### 1. 病例工作台（首页）
这是评委第一眼看到的页面，也是最重要的入口页面。

#### 页面目标
让用户意识到自己面对的是一个“待处理卒中病例任务”，而不是一个普通的上传工具。

#### 页面内容
- 病例头部信息：病例 ID、年龄、性别、检查时间、模态组合
- 当前状态：待分析 / 运行中 / 待复核 / 已完成
- 风险等级：高 / 中 / 低
- 当前任务目标：完成卒中首诊辅助分析
- 推荐路径提示：有无真实 CTP
- 启动按钮：启动 StrokeClaw 分析流程

### 2. 运行轨迹页（Agent Cockpit）
这是最能体现“智能体感”的核心页面。

#### 页面目标
让评委看到系统不是在后台黑盒运行，而是在一步一步做事。

#### 页面结构
- 左侧：`Stroke Orchestration` 时间线
- 中间：节点卡片流
- 右侧：关键摘要面板

#### 左侧时间线
展示完整步骤，并用状态区分：
- 未开始
- 运行中
- 已完成
- 风险中断
- 等待人工确认

#### 中间节点卡片
每张卡片展示：
- 节点名
- 状态
- 输入摘要
- 输出摘要
- 风险提示
- 下一步动作

#### 右侧摘要面板
展示：
- 当前阶段
- 已完成节点数
- 当前风险数
- 待确认项数量
- 当前关键结论摘要

### 3. 报告与证据页
该页负责把“最终结论”和“证据来源”绑定在一起。

#### 页面目标
让评委看到：系统不仅能写报告，还能尽量说明依据。

#### 页面内容
- 左侧：结构化关键指标
- 中间：自然语言报告草稿 / 最终报告
- 右侧：证据映射、规则支持、风险提示

### 4. 医生复核页
这是人机协同的关键页面。

#### 页面目标
让评委感受到医生在流程中是“正式参与者”，而不是最后“按一个确认键”的角色。

#### 页面内容
- 待确认项清单
- AI 建议内容
- 高风险提示
- 可编辑报告内容
- 操作按钮：
  - 一键采纳
  - 修改后采纳
  - 驳回建议
  - 标记低置信
  - 转交上级复核

### 5. 归档与回写页
这是任务闭环页面。

#### 页面目标
让评委看到：病例任务最终真正结束并沉淀为结果，而不是只停留在生成文本。

#### 页面内容
- 最终状态：已完成 / 已归档
- 最终报告版本
- 确认医生
- 回写状态
- 导出记录
- trace_id / run_id

### 6. 病例级 AI 问答（侧边抽屉）
AI 问答不应抢主流程，而应作为围绕当前病例上下文的辅助能力存在。建议以侧边抽屉或浮层方式出现，而不是独立首页。

## 四、前端交互应遵循的设计原则

### 1. 业务语言优先，技术语言作为第二层
第一层界面建议使用临床业务语言，如：
- 正在进行影像质控
- 正在生成类 CTP
- 正在分析缺血半暗带

第二层再显示对应技术节点名，如：
- `Image_QC.validate()`
- `MRDPM_Generate.run()`

这样既保留智能体感，也不让非技术评委一上来被工程术语吓退。

### 2. 渐进展开，而不是一次性展开全部信息
建议默认只展开：
- 当前运行节点
- 刚完成的上一个节点
- 风险节点
- 待人工确认节点

其他已完成节点折叠为摘要，防止页面拥挤。

### 3. 风险优先显示
所有高风险项、低置信项、待补录项应优先浮到前端界面中，不应埋在长报告末尾。

### 4. 强化“系统正在推进任务”的时间感
前端应通过状态变化、卡片滚动、节点高亮、阶段切换，让用户感到流程是在“推进”，而不是静态展示一组结果。

### 5. 旧入口保留，新入口增强
不应粗暴替换现有 viewer / report / chat，而应在其上新增 Agent Cockpit、任务流页面和归档闭环页，形成渐进式演进。

## 五、前端需要如何实现

前端实现上，最关键的是从“单次请求页面”转向“任务状态机页面”。

### 1. 以 `run_id` 为核心驱动前端
每次病例分析应被视为一次独立 run。前端流程为：

1. 创建 run
2. 获取 `run_id`
3. 基于 `run_id` 获取状态与事件
4. 渲染时间线与节点卡片
5. 在人工确认处提交动作
6. 获取最终结果与归档信息

### 2. 用状态机管理页面，而不是单个 loading 变量
建议前端至少维护以下总体状态：

- `queued`
- `running`
- `succeeded`
- `failed`
- `cancelled`

并维护阶段状态，例如：

- `intake`
- `qc`
- `routing`
- `generation`
- `analysis`
- `evidence`
- `review`
- `finalize`

### 3. 节点 UI 基于事件流渲染
节点卡片不应由前端手写固定流程，而应由后端返回的事件驱动，建议每个事件包含：

- event_id
- run_id
- phase
- node_name
- status
- started_at / ended_at
- input_summary
- output_summary
- risk_level
- requires_human_review
- evidence_refs

这样前端展示的“流程”，才真正对应后端真实运行过程。

### 4. 实时更新方式
建议分三档实现：

- **基础版**：轮询 `events`
- **增强版**：SSE
- **进阶版**：WebSocket

对于比赛演示，优先建议使用“轮询 + 节点顺序动画”，更稳定。

### 5. 核心前端组件建议
建议至少实现以下组件：

- `CaseHeader`
- `StepTimeline`
- `AgentNodeCard`
- `RiskAlertPanel`
- `EvidenceDrawer`
- `ReviewActionBar`
- `FinalizationPanel`

## 六、前端如何与后端底层架构配合

前端交互设计不是单独美化页面，而是对后端编排能力的“可视化翻译”。

### 1. 前端时间线，对应后端服务与编排层
前端的 `Stroke Orchestration` 映射的是后端的任务调度、状态机、风险中断、人工确认与异常处理。前端把这一层“看不见的运行逻辑”翻译为“看得见的流程”。

### 2. 前端节点卡片，对应后端 Agent / Skill
每张节点卡片应对应后端的真实执行对象，例如：

- `Image_QC.validate()` ← Image QC Skill
- `MRDPM_Generate.run()` ← MRDPM Generate Skill
- `Stroke_Analysis.segment()` ← Stroke Analysis Skill
- `Human_Review.await_action()` ← Human Review Skill

这样前端才不是“模拟流程”，而是“展示真实流程”。

### 3. 前端证据页，对应后端 Evidence 数据结构
报告与证据页应直接消费后端的 EvidenceItem / EvidenceMap 等结构，而不是前端手动拼接说明文字。

### 4. 前端人工确认，对应后端状态跳转
前端所有“采纳 / 修改 / 驳回 / 转交”动作都应驱动后端 run 状态变化，而不能仅停留在界面层。

### 5. 前端归档页，对应后端 FinalReport / Trace
最终页面展示的“归档、回写、版本、trace_id”应来自后端权威结果对象，而不是前端本地状态。

## 七、建议的分阶段实施方案

### 第一阶段：先做“能被评委感知到的 StrokeClaw”
重点完成：

- 病例工作台
- Stroke Orchestration
- 节点卡片
- 人工确认节点
- 闭环归档页

目标是让评委第一眼感受到，这不再是传统工具。

### 第二阶段：前后端打通
接入：

- 创建 run 接口
- 查询状态接口
- 查询事件接口
- 查询结果接口
- 人工确认动作接口

目标是让前端展示真正跟着系统运行，而不是纯静态 demo。

### 第三阶段：增加可信度与成熟度
补充：

- 证据抽屉
- 风险中断详情
- 版本记录
- Trace 回放
- 审计信息

目标是让 StrokeClaw 从“看起来像智能体”进一步升级为“可信的智能体系统”。

---

## 八、建议优先做出的 4 个前端亮点

若时间有限，建议优先做以下四项，因为它们最能打动评委：

### 1. 病例任务工作台
把“上传入口”改成“任务入口”。

### 2. Stroke Orchestration + 节点卡片
最直接体现“智能体在规划与执行”。

### 3. Human Review 节点
最能体现医疗场景下的人机协同成熟度。

### 4. 归档与回写页
最能把项目从“分析工具”拉升为“闭环系统”。

---

## 九、总结

对于 StrokeClaw 来说，真正打动评委的，不是后端多写了几个 Agent，也不是文档里写了多少架构层，而是前端是否让人真实感受到以下转变：

- 从“结果页”变成“任务页”
- 从“黑盒等待”变成“显式规划”
- 从“只看最终答案”变成“中间过程可见”
- 从“只报问题”变成“继续推进”
- 从“最后点确认”变成“正式人机协同”
- 从“报告输出完成”变成“病例闭环完成”

只要前端把这几个核心变化做出来，评委就会明显感知到：这不是原来的卒中先锋加了几个功能，而是真正开始呈现 **StrokeClaw** 的样子。
