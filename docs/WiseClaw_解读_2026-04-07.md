## 1) 核心定位

### 定位一句话
WiseClaw 可理解为 **面向医疗健康 ToB 场景的 Agent OS 平台**：以 `Agent + Skill` 为组织单元，把医疗 AI 从“单点问答能力”推进到“可集成、可治理、可运营的业务系统能力”。

### 产品角色、目标客户、行业问题
- 产品角色：医疗智能体运行与编排底座（而非单一模型 API）。
- 目标客户：医院、体检机构、药企/医学事务、保险与康养、健康管理平台等。
- 行业问题（对齐证据后）：
  - AI 能力可演示，但难进生产系统。
  - 跨系统对接、权限边界、合规审计、长期运营成本高。
  - 缺少可复用资产（工具、模板、知识与流程）沉淀机制。

## 2) 核心功能（按业务域）

### 能力清单
- 编排域：Agent + Skill 组装、任务流程化、业务自驱迭代。
- 证据域：报告/文档解析、结构化输出、可追溯解释链路（在公众号叙事中强调更强）。
- 治理域：多租户工作区隔离、权限与数据隔离、隐私/合规导向。
- 集成域：医疗数据与工具接入（官方提到医疗 MCP、业务数据、IoT）。
- 运营域：场景模板复用、低门槛上线与后续维护成本控制。

### 场景化能力（官网可见）
- 慢病管理 Agent
- 体检服务 Agent（检前建议 + 检后解读）
- 药品与特医饮食 Agent（院内核查 + 院外营养管理）
- Runtime 示例链路：`MedOCR -> MedVision AI -> ClinicalNLP -> HealthGraph -> RiskEngine -> TrendCompare -> ReportGen`

## 4) 技术架构（文字版）

### 架构图文字版
```text
[控制平面]
  - Gateway式统一入口（鉴权/路由/策略）
  - Agent/Skill 编排配置与任务分发
  - Workspace/租户隔离与权限策略

        |
        v

[执行平面]
  - 模型调用（WiseDiag 大模型）
  - 工具调用（Med-OCR、报告解读、体检分析、热量识别、健康档案等）
  - 任务链执行（chat/photo_read/calories 与上下文会话 topic_id）

        |
        v

[数据与合规平面]
  - 医疗知识与结构化数据
  - 业务系统/IoT 接入（官方称医疗 MCP）
  - 审计、追溯、隐私与合规控制（能力宣称）

        |
        v

[业务应用层]
  - 慢病管理 / 体检服务 / 药品与特医饮食等场景 Agent
```

### 关键对象（术语映射）
- `Agent`：面向任务的执行主体。
- `Skill`：可复用能力单元，支持组合。
- `Tool`：外部工具调用能力（OCR、解读、分析等）。
- `Session/topic_id`：多轮上下文管理（API 文档明确）。
- `Gateway`：统一接入与控制思想（OpenClaw 文档与公众号叙事一致；WiseClaw 内部实现未完全公开）。

## 5) 竞争与差异化（相对通用 Agent 平台）

- 医疗 ToB 取向更强：强调合规、隔离、审计、可运营，不只强调“生成效果”。
- 场景模板化：将医疗高频任务产品化（慢病、体检、药营）。
- 工具链偏医疗垂类：报告、OCR、体检分析、健康档案等能力组合。
- 接口兼容策略较友好：在部分能力上提供 OpenAI SDK 式接入，降低迁移成本。
- 与纯“通用模型 API”不同：其叙事中心是“行业 Agent 平台化交付”。

## 6) 结论分层

### 已证实事实
- WiseClaw 对外定位为医疗行业 Agent OS，核心范式为 Agent + Skill。
- 官网明确宣称工作区隔离、多租户、数据与工具打通（含医疗 MCP）。
- 官网给出慢病、体检、药品与特医饮食等场景化 Agent。
- API 文档明确存在：
  - 大模型接口（`wisediag-large-latest` + `/v1/chat/completions`）
  - 智能体工作流接口（`/v1/medicine/chat`，含 `task_name` 与 `topic_id`）
  - 多类工具能力与计费项（Med-OCR、报告类、体检类、热量识别等）
  - 医学向量模型（`ZZ-MedEmbedding-v1`）

### 高可信推断
- WiseClaw 在工程思想上借鉴/映射了 OpenClaw 的 Gateway 控制面与对象化架构（Gateway/Session/Tool/Node/Agent）。
- 其核心竞争力可能来自“医疗工具链 + 治理能力 + 场景模板”的组合，而非单一模型指标。

### 待验证项
- WiseClaw 内部运行时的真实部署拓扑、协议细节、故障恢复机制、审计数据模型。
- “物理级隔离”在不同部署模式下的具体实现边界。
- 与 OpenClaw 的代码级继承关系（目前公开材料多为叙事层，技术细节未完整披露）。

## 风险与不确定性

- 风险1：营销叙事与工程事实之间可能存在信息鸿沟。  
  应对：将“实现细节”与“能力宣称”分层，不混写。

- 风险2：公众号原文网页在抓取时存在编码与渲染限制。  
  应对：以原链接为准，并用可访问转载做交叉校验，不将不可复核片段纳入确定项。

- 风险3：图片分辨率较低，细节 OCR 不稳定。  
  应对：图片仅作观点线索，不作为关键事实主证据。
