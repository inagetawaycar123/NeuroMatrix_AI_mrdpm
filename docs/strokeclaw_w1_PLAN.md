## W1 细化方案（高借鉴 WiseClaw 风格版）

### Summary
- 周期：`2026-04-16 ~ 2026-04-19`。
- 目标：在“任务入口+计划可见”基础上，做出明显的科技感与交互推进感。
- 决策锁定：采用“高借鉴”交互风格；`底部 Agent Network 进度条` 纳入 W1 必做。
- 完成定义：用户进入 W1 页面后，能直观看到“任务编排-节点执行-状态推进”的过程感，而不是静态表单页。

### Key Changes
- 视觉体系冻结（借鉴交互语法，不做像素复刻）
  - 统一暗色仪表舱基底 + 青蓝/紫色信号色 + 风险红橙状态色。
  - 统一组件语言：`Orchestration 段落卡`、`节点执行卡`、`状态胶囊`、`底部网络条`。
  - 文案结构借鉴 WiseClaw：先“任务触发说明”，再“节点输入/输出”，最后“Result”。
  - 差异化约束：字体、圆角、间距、网格纹理、图标体系使用 StrokeClaw 自有 token，避免同质化 UI。

- W1 页面交互改造（前端）
  - 新增正式任务工作台 `/strokeclaw/tasks`，替代“直接上传即等待”感受。
  - 启动流程固定为两步：`计划预览` -> `确认执行`，禁止跳过预览直接跑。
  - `Stroke Orchestration` 面板支持节点分段显示，节点卡支持展开/收起详情。
  - 新增底部 `Agent Network` 固定栏：展示节点 chip、当前步高亮、总步数与进度。
  - 关键状态动效：步骤开始高亮、完成淡入勾选、Issue 警示闪烁一次、人工节点脉冲等待态。

- 数据与接口配合（W1）
  - `GET /api/strokeclaw/tasks`：任务工作台数据源。
  - `POST /api/agent/plans/preview`：仅预览计划，不创建 run。
  - `POST /api/agent/runs`：接收 `goal_question`（兼容 `question`）。
  - `GET /api/agent/runs/{run_id}` 与 `/events`：保证前端所需状态字段可直接映射到节点卡与网络条。

- 日程安排（按天）
  - `4/16`：冻结视觉 token、组件样式规范、页面 IA 与交互流程图。
  - `4/17`：完成工作台 + 计划预览面板 + 节点卡基础渲染。
  - `4/18`：接入真实接口与状态映射；完成底部 Agent Network 条与动效。
  - `4/19`：验收修复、演示脚本固化、输出 W1 验收记录。

### Public API / Interface Changes
- 新增 `GET /api/strokeclaw/tasks`。
- 新增 `POST /api/agent/plans/preview`。
- 增量兼容 `POST /api/agent/runs` 的 `goal_question`。
- 前端新增固定交互区域：`Agent Network Rail`（底部常驻，移动端折叠）。

### Test Plan（W1 验收标准）
1. 入口体验
- 从首页/上传页可进入 `/strokeclaw/tasks`，页面首屏出现任务卡 + 计划预览入口。

2. 交互链路
- 点击启动后必须先出现计划预览；预览失败时不得创建 run。
- 确认执行后创建 run 并进入 Runtime，状态可连续推进。

3. 视觉与交互质感
- 至少包含以下可见元素：Orchestration 卡、节点执行卡、状态胶囊、底部网络条。
- 状态切换有动效反馈（开始/完成/告警/等待），且不影响可读性。

4. 响应式与可用性
- Desktop 与 1366 宽度下布局不塌陷；移动端网络条可折叠。
- 键盘可聚焦主要按钮；错误态有明确文案，不白屏。

5. 回归安全
- `/upload -> /api/upload/start -> /processing` 主链路不回归。
- W0 联调页继续保持“仅 Mock”定位，不成为正式入口。

### Assumptions
- W1 仅聚焦“任务入口 + 计划可见 + 交互质感”，不包含 review 业务状态机实装。
- 视觉采用高借鉴方向，但坚持“交互模式借鉴、视觉资产自建”。
- 若性能压力出现，优先保留状态动效与网络条，降低装饰性背景动画。
