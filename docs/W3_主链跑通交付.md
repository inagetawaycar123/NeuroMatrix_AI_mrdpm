# W3 主链跑通交付（代码+文档冻结）

文档状态：Frozen (Week3)  
更新日期：2026-03-13  
边界：已实现 `triage -> tooling -> done`，本轮不含 ICV/EKV/Consensus

## 1. 交付摘要
1. 已落地规则版 `Triage Planner`，输入病例后可产出路径判定与固定 Tool 序列。
2. 已落地 `/api/agent/*` 最小接口集，可创建运行、查询状态/事件、拉取结果、失败重试。
3. 已落地失败节点重试与审计轨迹，重试遵循 `retryable + 次数上限`。

## 2. Triage Planner 协议（冻结）

### 2.1 PlannerInput
```json
{
  "run_id": "uuid",
  "patient_id": 1001,
  "file_id": "case_abc123",
  "available_modalities": ["ncct", "mcat", "vcat", "dcat"],
  "hemisphere": "right"
}
```

### 2.2 PlannerOutput
```json
{
  "imaging_path": "ncct_mcta",
  "tool_sequence": [
    "detect_modalities",
    "load_patient_context",
    "generate_ctp_maps",
    "run_stroke_analysis",
    "generate_medgemma_report"
  ],
  "should_generate_ctp": true,
  "should_run_stroke_analysis": true,
  "path_decision": {
    "raw_modalities": ["ncct", "mcat", "vcat", "dcat"],
    "canonical_modalities": ["ncct", "mcta", "vcta", "dcta"],
    "imaging_path": "ncct_mcta",
    "should_generate_ctp": true,
    "should_run_stroke_analysis": true
  }
}
```

### 2.3 判定规则（实现口径）
1. 历史别名统一归一化：`mcat/vcat/dcat -> mcta/vcta/dcta`。
2. 判定优先级：`ncct_mcta_ctp -> ncct_mcta -> ncct_single_phase_cta -> ncct_only`。
3. 仅 `ncct_mcta` 触发 `generate_ctp_maps`。
4. 仅 `ncct_mcta` 与 `ncct_mcta_ctp` 触发 `run_stroke_analysis`。
5. `hemisphere` 仅允许 `right|left|both`，非法值归一化为 `both` 并记录 warning。

## 3. 主链与 Tool 顺序（冻结）

| ImagingPath | 固定 Tool 顺序 |
|---|---|
| `ncct_only` | `detect_modalities -> load_patient_context -> generate_medgemma_report` |
| `ncct_single_phase_cta` | `detect_modalities -> load_patient_context -> generate_medgemma_report` |
| `ncct_mcta` | `detect_modalities -> load_patient_context -> generate_ctp_maps -> run_stroke_analysis -> generate_medgemma_report` |
| `ncct_mcta_ctp` | `detect_modalities -> load_patient_context -> run_stroke_analysis -> generate_medgemma_report` |

## 4. 重试/幂等/回滚（冻结）
1. 仅失败节点可重试，且必须 `retryable=true`。
2. 重试上限：`generate_ctp_maps=1`、`run_stroke_analysis=1`、`generate_medgemma_report=1`（在首次失败基础上可再试 1 次）。
3. 不回滚已成功步骤，重试从指定失败步骤继续执行到链路末端。
4. 同一 `run_id + step_key` 多次尝试保留事件轨迹（attempt），不覆盖历史记录。

## 6. 已知限制
1. Week3 运行态为内存态（进程重启后丢失）。
2. ICV/EKV/Consensus 尚未接入（按周计划在后续阶段实现）。
3. 若本地缺失模型/依赖文件，相关 Tool 会返回 `TOOL_DEPENDENCY_MISSING`。

## 7. Week3补全（可见化验收版，2026-03-13）

### 7.1 前端可见变化
1. 上传页新增 `Agent 主链` 开关（默认开启）：`#agentRunToggle`。
2. 上传请求在开关开启时附带 `start_agent_run=true`。
3. 上传成功后，若后端返回 `agent_run_id`，会透传到 `/processing?...&agent_run_id=...`。
4. processing page shows Agent summary card: `run status / stage / current tool / event count / report status / last error`.
5. Show `Retry failed step` only when `run.status=failed` and there is a step with `status=failed && retryable=true`; then call `/api/agent/runs/{run_id}/retry`.
6. Left step-6 (AI structured report) maps from Agent status when upload step is `skipped`: running/completed/failed.

### 7.2 终端可见变化
1. 新增统一 `[AGENT]` 日志前缀。
2. 日志字段至少包含：`run_id stage tool attempt status error_code latency_ms`。
3. 覆盖日志类型：`run_start`、`triage_planner completed/failed`、`tool completed/failed`、`retry_queued`、`run_done/run_failed`。

### 7.3 兼容性口径
1. 不传 `start_agent_run` 时，旧上传链 `/api/upload/start + /api/upload/progress/{job_id}` 保持不变。
2. 本轮不新增路由，仍使用既有 `/api/agent/*` 五个接口。
