# Week6 汇总与证据链交付

更新时间：2026-03-23  
状态：Implemented (Pending full clinical sign-off)

## 1. 目标与范围
1. 在不改 ICV/EKV/Consensus 判定算法语义的前提下，新增 Week6 汇总层能力。  
2. 统一以 `report_payload` 作为单一真值，补齐：
   - `final_report`
   - `evidence_items`
   - `evidence_map`
   - `traceability`
3. 不新增 API 路由，不改变现有 `/api/agent/runs*`、`/api/validation/context`、`/api/upload/*` 协议形态。

## 2. 已落地实现
1. 后端新增汇总器：`backend/summary_assembler.py`
   - 输入：`icv/ekv/consensus + report_payload + run/file context`
   - 输出：Week6 四个对象写回 `report_payload`
   - 日志：新增 `[SUMMARY]`、`[EVIDENCE]` 关键行（包含 coverage、mapped/total、high-risk unmapped）
2. Agent summary 阶段接入
   - 在 `generate_medgemma_report` 工具后，将汇总结果写入返回的 `report_payload`
   - `run.result.report_result.report_payload` 可直接读取 Week6 对象
3. 审计事件增强
   - `AgentEvent` 补充 `stage` 与 `event_seq`
   - `events` 可按 `event_seq` 回放
4. Validation 聚合接口增强
   - `/api/validation/context` 新增 `traceability`
   - 保持来源优先级：`run result > case payload`
5. 前端展示增强
   - Validation：新增 traceability 摘要卡（status、coverage、mapped/total、unmapped、高风险未映射）
   - Report：新增 “Evidence Traceability” 模块，展示关键结论与证据引用/不可用原因

## 3. Week6 契约（冻结）
### 3.1 `report_payload.final_report`
- `summary`
- `key_findings[]`
- `risk_level`
- `confidence`
- `citations[]`
- `uncertainties[]`
- `next_actions[]`

### 3.2 `report_payload.evidence_items[]`
- `evidence_id`
- `source_type`
- `source_ref`
- `claim`
- `claim_id`
- `support_level`
- `timestamp`
- `snippet`
- `doc_name`
- `page`
- `run_id`
- `file_id`

### 3.3 `report_payload.evidence_map`
- `finding_id -> { evidence_ids[], unavailable_reason }`

### 3.4 `report_payload.traceability`
- `total_findings`
- `mapped_findings`
- `coverage`
- `unmapped_ids[]`
- `high_risk_unmapped_count`

## 4. 可追溯判定规则
1. 关键结论默认覆盖 6 类 claim：
   - `hemisphere`
   - `core_infarct_volume`
   - `penumbra_volume`
   - `mismatch_ratio`
   - `significant_mismatch`
   - `treatment_window_notice`
2. 允许追加 ICV 高风险 finding 作为补充结论。
3. 每条结论必须满足其一：
   - 有 `evidence_ids`
   - 有 `unavailable_reason`
4. 高风险结论（core/penumbra/mismatch/significant_mismatch）必须可追溯或有明确 unavailable 原因。

## 5. 兼容策略
1. 旧前端仅依赖原 `report_payload` 字段时不受影响。  
2. 新增 Week6 字段缺失时，前端按 unavailable 回退，不抛异常。  
3. 不做 `run.result` 顶层双写，避免双源漂移。

## 6. 验收阈值（Week6）
1. `traceability.coverage >= 0.90`（建议阈值）  
2. 高风险结论 100% 可追溯或明确 unavailable 原因  
3. `events` 可观察 `icv -> ekv -> consensus/skip -> summary -> done`
