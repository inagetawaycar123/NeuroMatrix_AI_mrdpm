# W6 汇总与证据链交付文档

## 1. 概述

Week6 的核心目标是将 Week5 的 ICV/EKV/Consensus 结果"收口成可审计结论"，实现 `tooling -> icv -> ekv -> consensus -> summary -> done` 链路中的 `summary` 环节，产出 **FinalReport + EvidenceMap + Trace**。

## 2. 实现状态

### 2.1 已完成的功能

1. **契约冻结**：
   - `FinalReport` 数据结构：包含 `summary`、`key_findings`、`risk_level`、`confidence`、`citations`、`uncertainties`、`next_actions`、`traceability`
   - `EvidenceItem` 数据结构：包含 `evidence_id`、`source_type`、`source_ref`、`claim`、`support_level`、`timestamp`、`snippet`、`doc_name`、`page`、`run_id`、`file_id`
   - `AgentEvent` 数据结构：已包含 `stage` 与 `event_seq` 字段

2. **后端汇总器**：
   - 新增 `backend/summary.py` 模块，实现 `build_final_report(run)` 函数
   - 支持从 `tool_results + report_payload + ekv/consensus` 生成 `FinalReport`
   - 关键结论集合固定为 6 类 claim：
     - `hemisphere`（偏侧）
     - `core_volume_ml`（核心体积）
     - `penumbra_volume_ml`（半暗带体积）
     - `mismatch_ratio`（不匹配比）
     - `significant_mismatch_present`（显著不匹配）
     - `treatment_window_hint`（时窗提示）
   - 生成 `evidence_map`：每条 `key_finding` 必须附 `evidence_ids[]`
   - 计算 `traceability`：`total_findings`、`mapped_findings`、`coverage`、`unmapped_ids`
   - 写入策略：`run.result` 增加 `final_report`、`evidence_items`、`evidence_map`、`trace`

3. **审计链收口**：
   - 统一事件写入点：所有工具开始/完成/失败都带 `stage` 快照
   - `events` 输出保证时序可回放（`event_seq` 单调递增）
   - 日志增加 `[SUMMARY]`、`[EVIDENCE]` 关键行

4. **前端可见化**：
   - Processing 页面右侧摘要显示 ICV/EKV/Consensus 状态和发现数量
   - Viewer 页面有 ICV 静态面板，显示 ICV 校验结果
   - Viewer 页面支持从 `report_payload` 提取 ICV/EKV/Consensus 数据并生成 HTML 展示
   - 前端有 `buildIcvSummaryHtml`、`buildEkvSummaryHtml`、`buildConsensusSummaryHtml` 函数

5. **测试覆盖**：
   - `tests/test_summary.py`：10 个测试用例，全部通过
   - `tests/test_ekv.py`：3 个测试用例，全部通过
   - `tests/test_consensus.py`：3 个测试用例，全部通过
   - `tests/test_icv_rules.py`：3 个测试用例，全部通过

### 2.2 缺口与待完善

1. **前端证据链可视化**：
   - 目前前端有 ICV 静态面板和 ICV/EKV/Consensus HTML 构建函数

## 3. 契约定义

### 3.1 FinalReport 数据结构

```python
{
    "summary": str,  # 报告摘要
    "key_findings": List[Dict],  # 关键发现列表
    "risk_level": str,  # 风险等级
    "confidence": float,  # 置信度
    "citations": List[Dict],  # 引用列表
    "uncertainties": List[str],  # 不确定性列表
    "next_actions": List[str],  # 下一步行动
    "traceability": {  # 可追溯性信息
        "total_findings": int,
        "mapped_findings": int,
        "coverage": float,
        "unmapped_ids": List[str]
    }
}
```

### 3.2 EvidenceItem 数据结构

```python
{
    "evidence_id": str,  # 证据ID
    "source_type": str,  # 来源类型（icv/ekv/consensus）
    "source_ref": str,  # 来源引用
    "claim": str,  # 支持的claim
    "support_level": str,  # 支持级别（supported/partially_supported/not_supported）
    "timestamp": str,  # 时间戳
    "snippet": str,  # 证据片段
    "doc_name": str,  # 文档名称
    "page": int,  # 页码
    "run_id": str,  # 运行ID
    "file_id": str  # 文件ID
}
```

### 3.3 AgentEvent 数据结构

```python
{
    "event_id": str,  # 事件ID
    "run_id": str,  # 运行ID
    "stage": str,  # 阶段
    "event_seq": int,  # 事件序列号
    "event_type": str,  # 事件类型
    "timestamp": str,  # 时间戳
    "payload": Dict  # 事件负载
}
```

## 4. 映射规则

### 4.1 关键结论到证据的映射

1. **NCCT-only 场景**：
   - CTP 类结论为 `unavailable`，必须有原因字段
   - 证据来源主要为 ICV 校验结果

2. **NCCT+mCTA 场景**：
   - 每条关键结论均有 `evidence_ids` 或明确 unavailable 原因
   - 证据来源包括 ICV 和 EKV

3. **NCCT+mCTA+CTP 场景**：
   - 量化结论与 EKV citations 可对齐到 `doc+page`
   - 证据来源包括 ICV、EKV 和 Consensus

### 4.2 证据构建逻辑

1. **ICV 证据**：从 ICV findings 中提取，支持级别根据 status 判断
2. **EKV 证据**：从 EKV claims 中提取，包含文献引用信息（doc_name, page）
3. **Consensus 证据**：从 Consensus 决策中提取，包含冲突解决信息

## 6. DoD（Definition of Done）

### 6.1 核心 DoD

**"报告中的每条关键结论都能追溯到至少 1 条证据，或明确标注 unavailable + 原因"**

### 6.2 具体验收标准

1. **证据映射必测**：
   - NCCT-only：CTP 类结论为 `unavailable`，且必须有原因字段
   - NCCT+mCTA：每条关键结论均有 `evidence_ids` 或明确 unavailable 原因
   - NCCT+mCTA+CTP：量化结论与 EKV citations 可对齐到 `doc+page`

2. **一致性必测**：
   - 同一 `file_id` 下，Processing/Viewer/Report 的 `finding_count`、`conflict_count`、`traceability` 一致
   - `events` 可按 `event_seq` 回放，阶段出现 `icv -> ekv -> consensus/skip -> summary`

3. **非阻断回归**：
   - 强制 EKV 或 Consensus 失败，run 仍可 `succeeded`，但 `final_report.uncertainties` 必须包含失败原因
   - Agent 关闭路径不变，上传-处理-查看链路不退化

4. **签收阈值**：
   - `traceability.coverage >= 0.9`（建议阈值）
   - 高风险结论（核心体积/半暗带/不匹配）必须 100% 可追溯或明确 unavailable
