# 卒中先锋如何向 WiseClaw 借鉴：从“专病应用”升级为“StrokeClaw”

### ——面向急性缺血性卒中首诊场景的专病智能体辅助决策系统设计方案

## 一、项目定位与核心判断

卒中先锋最值得向 WiseClaw 借鉴的，不是去做一个“大而全”的医疗平台，也不是简单模仿其界面形式，而是学习其将 AI 从“单点功能”升级为“任务运行底座”的方式。当前卒中先锋已经具备一条较完整的卒中辅助诊疗主链路，包括影像输入、MRDPM 生成 class-CTP、卒中自动分析、结构化报告生成、医生复核、AI 问答与结果留痕。在此基础上，项目下一步更合理的方向，不是演化为通用医疗 Agent 平台，而是升级为一个面向卒中场景的专病 Agent OS。

**StrokeClaw = 面向卒中场景的专病智能体运行底座。**

它不以“一次性输出结论”为目标，而是围绕一个卒中病例，组织多个智能体、多个技能和多个关键状态节点，分阶段完成：

- 病例接收
- 影像质控
- 模态判断
- class-CTP 生成
- 卒中自动分析
- 证据与规则校验
- 报告生成
- 医生复核
- 回写归档

因此，StrokeClaw 的本质不是“一个更复杂的卒中分析工具”，而是一套围绕病例任务展开的、可规划、可执行、可追踪、可验证、可协同、可闭环的专病智能体系统。

---

## 二、为什么卒中场景适合发展为 StrokeClaw

卒中首诊天然不是一个“单问单答”问题，而是一个具有强流程特征、强时效特征和强协同特征的临床任务。医生在真实工作中，不只是需要一个“诊断结果”，而是需要完成一整条连续流程：

1. 判断病例与影像信息是否完整  
2. 判断影像是否可用、是否存在关键缺失  
3. 判断是否存在真实 CTP  
4. 若无真实 CTP，是否能够通过替代性方法获取关键灌注信息  
5. 基于影像与临床信息完成核心梗死、半暗带与 mismatch 分析  
6. 根据证据和规则形成可解释结论  
7. 高效完成结构化报告与文本表达  
8. 在关键节点引入医生确认  
9. 形成可回写、可归档、可审计的最终结果

这说明卒中场景并不适合停留在“上传影像—输出结果”的工具模式，而更适合发展为一个围绕病例任务展开的专病智能体系统。StrokeClaw 的意义，就在于把卒中先锋当前已经具备的能力，从“结果输出工具”升级为“流程推进系统”。

## 三、StrokeClaw 对 WiseClaw 的借鉴边界

StrokeClaw 向 WiseClaw 借鉴的重点，不是做一个覆盖所有医疗场景的通用平台，而是借鉴其更底层的方法论与系统组织方式。这个借鉴应当是“取方法，不照搬定位”。

### 1. 借鉴的是系统化方法，而不是业务边界
WiseClaw 更偏平台层，服务多场景、多任务、多角色；而 StrokeClaw 应坚持聚焦卒中单病种，尤其聚焦急性缺血性卒中首诊这一高价值场景。

### 2. 借鉴的是运行底座思维，而不是大而全思维
StrokeClaw 不应追求变成一个泛医疗 Agent 平台，而应在卒中场景中形成自己的“专病运行底座”。

### 3. 借鉴的是任务流交互方式，而不是单纯页面样式
真正值得学习的不是某一个按钮或卡片设计，而是“先规划、再执行、途中可见、关键处确认、最终闭环”的任务型交互逻辑。

### 4. 借鉴的是人机协同机制，而不是无边界自动化
StrokeClaw 必须明确：它是辅助医生，而不是替代医生。所有高风险结论、关键治疗建议和最终报告确认，都应留有正式的人机协同节点。

## 四、StrokeClaw 的总体设计目标

在卒中先锋现有基础上，StrokeClaw 的设计目标应当从“做出功能”进一步升级为“构建系统能力”。建议明确形成以下六个目标：

### 1. 可规划
系统在执行前能够明确展示病例处理计划，让用户知道接下来会做什么，而不是在后台黑盒运行。

### 2. 可执行
每一步都由明确的 agent 或 skill 承担，形成完整的病例处理任务链。

### 3. 可追踪
关键中间结果、状态变化、风险提示、人工操作和最终版本都可以被记录和追踪。

### 4. 可验证
关键结论能够绑定影像指标、规则、证据来源和人工确认信息，而不是只给出最终答案。

### 5. 可协同
系统在关键风险节点主动请求人工介入，形成真实的人机协同，而非伪全自动。

### 6. 可闭环
最终结果能够形成可导出、可回写、可归档、可审计的闭环流程。

## 五、StrokeClaw 在卒中先锋基础上应新增的系统能力

当前卒中先锋已经具备较强的卒中主链路能力。StrokeClaw 的升级重点，不是简单增加功能，而是增加“系统层能力”。

### 1. 病例任务管理能力
将每个病例从一次性输入升级为任务对象，至少包含：

- 病例 ID
- 患者基础信息
- 当前模态组合
- 当前处理阶段
- 风险级别
- 当前责任节点
- 待确认事项
- 当前版本状态

这样，系统面对的就不再是一组待分析数据，而是一个具有生命周期的“病例任务”。

### 2. 显式编排能力
在交互层增加 `Stroke Orchestration`，让用户在运行前就知道本次病例将经历哪些步骤，而不是静默等待系统给出结果。

### 3. 中间结果显性化能力
将关键中间结果从后台计算结果转变为可视化展示对象，例如：

- 影像序列检测结果
- 质控结果
- 模态路由结果
- MRDPM 生成状态
- core / penumbra / mismatch 等关键指标
- 风险提示与低置信提示
- 规则与证据校验结果

### 4. 风险中断与推进能力
当系统发现问题时，不只提示错误，而是继续推动流程向前，例如提示补录、建议复核、生成待补充版报告、降低置信度等。

### 5. 正式人机协同能力
把医生确认节点从“普通按钮”升级为显式任务节点，如 `Human_Review.await_action()`，并明确待确认项、风险项、AI 建议和可选动作。

### 6. 审计与版本能力
记录从输入到回写的完整过程，包括：

- 节点调用记录
- 中间结果快照
- 风险提示记录
- 医生修改记录
- 版本变更记录
- 最终归档记录

这些能力共同决定了 StrokeClaw 不是一个“会生成结果的应用”，而是一套“能运行病例任务的系统”。

## 七、StrokeClaw 的交互设计原则：让用户感受到“智能体在做事”

WiseClaw 的强交互感，不是来自聊天语气，而是来自“任务被一步一步推进”的体验。StrokeClaw 若要让用户真正感受到智能体在规划后为其逐步完成工作，应重点遵循以下交互原则。

### 1. 有任务入口，而不是直接给答案
用户进入系统后，先看到“待处理卒中病例”，再启动流程，而不是上传后直接等待结果。

### 2. 有执行计划，而不是后台默默运行
先展示 `Stroke Orchestration`，明确告诉用户接下来将执行哪些步骤。

### 3. 有节点卡片，而不是一个大黑盒面板
每一步都应有独立节点，例如：

- `Image_QC.validate()`
- `Modality_Router.route()`
- `MRDPM_Generate.run()`
- `Stroke_Analysis.segment()`
- `Evidence_Check.verify()`
- `Human_Review.await_action()`
- `EMR_Sync.writeback()`

### 4. 有中间结果，而不是只给最终结论
用户应看到每一步“理解成了什么”“发现了什么”“下一步要做什么”。

### 5. 有状态变化，而不是平铺输出
建议统一使用清晰状态标签：

- Running
- Completed
- Warning
- Issue Found
- Await Review
- Finalized

### 6. 有人工介入断点，而不是假装全自动
关键临床节点必须明确停下来，等待医生确认，而不是把人工确认弱化为最终页面上的一个按钮。

### 7. 有任务闭环，而不是“报告生成完毕”
最终结束语不应只是“结果已生成”，而应是：

- 病例处理完成
- 医生已确认
- 报告已归档
- 结果已回写 / 可导出 / 可进入随访

这样，系统给用户的感受就会从“AI 给了我一份结果”，转变为“一个卒中智能体替我推进并完成了病例处理任务”。

## 八、推荐的 StrokeClaw 交互链路

### Stroke Orchestration
1. Case Intake  
2. Image QC  
3. Modality Routing  
4. MRDPM Generation  
5. Stroke Analysis  
6. Evidence & Rule Check  
7. Structured Conclusion  
8. Report Drafting  
9. Human Review  
10. Writeback / Archive  

### 对应节点
- `Case_Intake.parse()`
- `Image_QC.validate()`
- `Modality_Router.route()`
- `MRDPM_Generate.run()`
- `Stroke_Analysis.segment()`
- `Evidence_Check.verify()`
- `Report_Generate.compose()`
- `Human_Review.await_action()`
- `EMR_Sync.writeback()`

### 关键中间结果示例

#### `Image_QC.validate()`
- 检测到序列：NCCT、动脉期 CTA、静脉期 CTA、延迟期 CTA
- 质控结果：通过 / 警告 / 失败
- 风险提示：运动伪影、序列缺失、重建异常

#### `Modality_Router.route()`
- 检测结果：有无真实 CTP
- 路由决定：进入真实 CTP 路径或 MRDPM 路径

#### `MRDPM_Generate.run()`
- 输出：CBF / CBV / Tmax
- 状态：Completed
- 可信度提示：高 / 中 / 低

#### `Stroke_Analysis.segment()`
- 梗死核心体积：xx ml
- 半暗带体积：xx ml
- mismatch ratio：xx
- 风险提示：存在明显可挽救组织 / 边界不清晰

#### `Human_Review.await_action()`
- 待确认项：core / penumbra / mismatch 结论
- 高亮项：缺失信息、高风险项、低置信项
- 可选动作：采纳、修改、驳回、转交复核

这一交互链路的核心目的，是让用户清楚感受到：系统在规划、执行、发现问题、请求协同并最终完成任务闭环。

## 九、StrokeClaw 的最小可行版本（MVP）

为了让 StrokeClaw 从概念走向可实现，建议先定义一版明确的 MVP。

### MVP 目标
优先跑通最有临床价值、最有智能体感的主链路：

**病例任务启动 → 影像质控 → 模态路由 → MRDPM 生成 → 卒中分析 → 结构化结论 → 医生复核 → 最终归档**

### MVP 必须具备的 5 个页面
1. 病例工作台  
2. 执行轨迹页  
3. 结果与证据页  
4. 医生复核页  
5. 最终归档页  

### MVP 必须具备的 5 个核心节点
1. `Image_QC.validate()`  
2. `Modality_Router.route()`  
3. `MRDPM_Generate.run()`  
4. `Stroke_Analysis.segment()`  
5. `Human_Review.await_action()`  

### MVP 必须达成的用户感知
- 我知道系统要做什么
- 我知道系统当前做到哪一步
- 我知道系统为什么给出这个结论
- 我知道哪些地方还需要我确认
- 我知道病例最终如何闭环完成

## 十、实施路径建议

### 第一阶段：主链路显性化
先不急于增加更多功能，而是把现有卒中先锋主链路可视化、任务化：

- 输入
- 路由
- 生成
- 分析
- 报告
- 复核
- 回写

### 第二阶段：主链路 Skill 化
将主链路中的关键步骤正式抽象为 Stroke Skills，形成可编排、可复用的专病能力库。

### 第三阶段：交互任务流化
新增：

- 病例工作台
- Runtime View
- 风险中断节点
- Human Review 节点
- 回写与归档页

### 第四阶段：面向全流程扩展
在首诊链路稳定后，再逐步扩展至：

- 转诊建议
- 出院管理
- 二级预防
- 康复随访

这样 StrokeClaw 才能从“卒中分析工具”逐步演进为“卒中全流程专病智能体系统”。

## 十一、如何评估 StrokeClaw 是否真正具备“智能体感”

为了避免 StrokeClaw 只停留在概念设计层，建议建立一套评估维度。

### 1. 交互评估
- 用户是否能看懂当前执行到哪一步
- 用户是否能看到关键中间结果
- 用户是否能快速定位待确认项
- 用户是否感受到系统是在推进任务而不是一次性吐答案

### 2. 工程评估
- 每个节点是否有明确状态机
- 是否支持风险中断与失败重试
- 是否记录版本与人工操作
- 是否支持审计与追踪

### 3. 临床辅助评估
- 是否减少报告补录次数
- 是否减少关键字段遗漏
- 是否缩短报告生成与复核时间
- 是否提升结构化结论与最终报告的一致性

### 4. 智能体特征评估
- 是否存在显式规划
- 是否存在多节点协同
- 是否存在主动风险提示
- 是否存在正式人工确认
- 是否形成最终任务闭环

## 十二、风险边界与产品边界

### StrokeClaw 不是什么
- 不是替代医生的自动诊断系统
- 不是覆盖全医疗场景的通用平台
- 不是完全依赖自由生成的黑盒系统
- 不是无人工确认的全自动执行器

### StrokeClaw 是什么
- 是面向卒中首诊场景的专病智能体辅助决策系统
- 是一个以病例任务为中心的运行底座
- 是一套可规划、可执行、可追踪、可验证、可协同、可闭环的专病 Agent OS

这种边界清晰的定位，有助于评委理解项目的创新点，也有助于突出项目在医疗高风险场景中的谨慎性与可落地性。

## 十三、总结

WiseClaw 给卒中先锋最大的启发，不是去做一个更大的医疗平台，也不是简单模仿一个“会跑流程的界面”，而是让我们意识到：真正可落地的医疗智能体，不应只是单点输出工具，而应是一套围绕病例任务展开、可规划、可执行、可追踪、可验证、可协同、可闭环的运行底座。

基于这一认识，卒中先锋未来更合理的升级方向，是在现有卒中主链路基础上，逐步演进为面向卒中首诊场景的专病 Agent OS——**StrokeClaw**。它既保留卒中先锋在卒中专病上的深度，又吸收 WiseClaw 在系统组织、任务流交互和人机协同方面的方法论优势，从而形成一个更完整、更可信、更有“智能体感”的卒中辅助决策系统。
