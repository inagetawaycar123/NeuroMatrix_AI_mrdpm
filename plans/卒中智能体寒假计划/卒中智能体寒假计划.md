# 卒中智能体寒假计划（可直接落地执行版，无具体时间表）

本文严格对齐寒假计划要求：[`plans/卒中智能体寒假计划.docx`](plans/卒中智能体寒假计划.docx)，并与现有工程能力保持一致（系统能力概览见：[`docs/CORE_FUNCTIONS.md`](docs/CORE_FUNCTIONS.md)）。

计划总目标是在现有 Web 系统基础上稳定演示并端到端跑通四种影像模态组合的上传、展示、自动编排、卒中分析、报告生成与对话问答闭环，四种组合为 NCCT-only、NCCT + 单期 CTA、NCCT + mCTA（三期）、NCCT + CTA + CTP 参数图。核心原则是 NCCT 必选、多模态可选，缺失模态与质量差数据必须走可解释的降级路径，同时不破坏现有系统入口与能力（上传、viewer、对比度调节、伪彩生成、手动卒中分析、报告保存）。

---

## 1. 项目背景与目标

现有系统已具备端到端卒中影像工作台能力（上传与viewer、MRDPM/Palette 灌注生成、卒中分析量化、报告保存与查看），参考：[`docs/CORE_FUNCTIONS.md`](docs/CORE_FUNCTIONS.md)。寒假新增目标是把系统升级为“卒中智能体”形态：医生录入基本信息并上传自己拥有的影像模态，系统自动识别组合并自动编排，按模态分别调用本地部署 Medgemma 1.5 生成每模态独立报告段落与结构化结论，再融合 B 的卒中量化证据生成总报告；同时对话侧新增按病人 ID 直读报告并脱敏问答能力。

必须满足的业务要求来自 [`plans/卒中智能体寒假计划.docx`](plans/卒中智能体寒假计划.docx)：模态组合更贴近临床；使用本地 Medgemma 1.5 做按模态独立分析与自动报告（替代原“百川M3不读片”的局限）；若满足条件自动进行卒中分析（不依赖手动按钮）；RAG 支持医生输入病人 id 直读报告并脱敏后问答；前端动态展示医生上传与系统生成的全部影像产物，对比度调节与伪彩功能不受影响。

## 2. 范围与非目标

范围内（必须交付）：

1) 四种组合的自动识别与自动编排入口，且不破坏现有入口与旧流程，后端编排入口以 [`app.py`](app.py) 为主，上传入口为 [`upload_files()`](app.py:2803)，viewer入口为 [`viewer_page()`](app.py:2798)。

2) 卒中分析自动化：无论 CTP 参数图来源是 MRDPM 生成还是医生上传真实 CTP 参数图，都必须复用当前 web 端卒中分析逻辑（半暗带、核心梗死、不匹配等），核心算法在 [`class StrokeAnalysis`](stroke_analysis.py:16) 与 [`analyze_stroke_case()`](stroke_analysis.py:374)。

3) 报告自动化：按模态独立调用 Medgemma 1.5 输出结构化结果与模态段落，并把卒中量化结果作为证据注入报告。模态输出边界在本文内固定为：NCCT 重点为出血排除、早期缺血征象、ASPECTS 倾向；CTA/mCTA 重点为 LVO 部位与侧支提示；CTP 重点为核心/半暗带/不匹配与 DEFUSE3 或 DAWN 风格量化提示（其中体积与不匹配等关键数值必须来自卒中量化模块，不允许由模型凭空生成）。

4) viewer 动态展示：根据实际上传组合与生成产物动态呈现影像与结果卡片；已存在的对比度调节功能与伪彩生成入口必须保持可用（伪彩链路参考 [`generate_all_pseudocolors_route()`](app.py:895)）。

5) RAG 对话按 patient_id 直读并脱敏：在现有对话基础上新增入口，读取该 patient_id 的最新报告与结构化关键字段，脱敏后回答并可追溯引用；现有对话入口参考 [`api_chat_clinical_stream()`](app.py:1443) 与 [`api_chat_clinical()`](app.py:1562)。

非目标（本计划不以此作为必验收口径）：

1) 不以“Medgemma 读片准确率”作为验收指标，验收只看可解析、可追溯、可降级、可演示；临床效果评测与指标看板属于后续阶段。

2) 不引入新的复杂前端框架重写 viewer（现有 viewer 以模板与原生 JS 为主，见 [`templates/patient/upload/viewer/index.html`](templates/patient/upload/viewer/index.html) 与 [`static/js/viewer.js`](static/js/viewer.js)），寒假只做最小侵入式扩展。

3) 不强制引入异步队列/分布式任务系统；可先同步实现编排与阶段状态，后续再替换为异步实现但接口契约保持不变。

## 3. 整体端到端流程（四组合覆盖，从上传到viewer到分析到报告到对话）

统一流程（不依赖医生手动点按钮）：

1) 医生提交病患基本信息，沿用现有病人信息录入与入库接口（插入逻辑见 [`insert_patient_info()`](app.py:30)，接口见 [`api_insert_patient()`](app.py:1022)）。

2) 医生上传当前拥有的影像数据，NCCT 必选，其余按实际上传；上传入口仍为 [`upload_files()`](app.py:2803)，前端上传脚本为 [`processFiles()`](static/js/upload.js:72)。

3) 上传后后端生成 `case_manifest.json`（病例清单），并在上传响应里回传 `available_modalities` 与 `qc_summary`（可以暂时不管这个字段），前端据此识别组合并进入 viewer。

4) 后端自动触发编排任务，编排接口固定为三件套 `POST /api/agent/run`、`GET /api/agent/status`、`GET /api/agent/result`（实现位置在 [`app.py`](app.py)），内部按 manifest 分流执行：

   4.1) NCCT 模态独立分析（Medgemma 1.5，输出 NCCT 段落与结构化结论）

   4.2) CTA/mCTA 模态独立分析（Medgemma 1.5，输出 CTA 段落与结构化结论）

   4.3) CTP 模态独立分析（Medgemma 1.5，输出 CTP 段落与结构化结论），但核心/半暗带/不匹配等量化必须来自卒中分析模块（B 的职责）

   4.4) 卒中分析自动触发（无论 CTP 来源于 MRDPM 或外部上传），复用 [`analyze_stroke_case()`](stroke_analysis.py:374) 的量化与可视化产出

   4.5) 汇总器拼装多模态总报告：把每模态段落 + 卒中量化证据统一拼装为 `report_draft_v1.md`，并可保存为 `report_final_v1.md`（医生可以编辑）（保存接口可复用或扩展 [`api_save_report()`](app.py:1276)）

5) viewer 动态展示：根据 `case_manifest.json` 与模块结果 JSON，动态展示“上传的原始影像”“系统生成灌注图”“卒中分析叠加图”“模态结论卡片”“状态机”，同时保留原有对比度调节与伪彩入口不受影响。

6) 对话问答：医生输入 patient_id，系统读取该 patient_id 关联的最新报告与结构化字段，脱敏后进入检索与回答，返回答案并引用来源片段与证据。

## 4. 团队分工与接口协作方式（统一契约、最小侵入、可回滚）


前后端展示协议：

1) viewer 不允许自行猜测“应该有什么模态”，必须完全基于 manifest 的 `available_modalities` 与 `/api/agent/status` 的模块状态渲染。

2) 缺失模态文案是固定模板，但触发条件必须由 manifest 与模块状态驱动。（不写或者写无）

3) 证据图片 URL 必须通过现有静态读取路由访问，优先复用 [`get_image()`](app.py:2934) 与 [`get_stroke_analysis_image()`](app.py:1006)，避免让前端直接拼本地磁盘路径。

### 4.2 协作方式

1) A 负责冻结与维护“契约三件套”：`case_manifest.json` 字段、模块结果 envelope v1、`/api/agent/*` API 字段与错误码。

2) B 负责维护“卒中分析输入输出契约”：外部CTP与MRDPM生成CTP如何映射到同一分析 pipeline，输出 `stroke_v1.json` 的字段口径与可视化产物路径。

3) C 负责维护“Medgemma按模态独立分析契约”：每模态 prompt 版本、输出 JSON 的可解析字段、与报告模板的字段映射。

4) D 负责维护“patient_id直读对话契约”：patient_id 到报告与结构化字段的检索路径、脱敏规则、引用策略与权限错误提示。

5) Git 协作采用 main + feature 分支，涉及 [`app.py`](app.py) 的改动必须经 A code review；涉及 viewer 的改动必须经 A 与 B/C/D 至少一人 review，避免接口字段变化无人知晓。

1) 输入目录约定：`static/processed/<file_id>/inputs/ctp_maps/`（外部）或 `static/processed/<file_id>/perfusion/generated/`（生成）。

2) 输出 JSON：`static/processed/<file_id>/module_results/stroke_v1.json`，包含 `core_volume_ml/penumbra_volume_ml/mismatch_ratio/threshold_basis/visuals.overlays[]`。

1) 无 CTP 时不报错但输出 `stroke` 模块为 skipped，给出建议补采；

2) 读取失败或生成失败时返回 `ERR_STROKE_ANALYSIS` 或 `ERR_PERFUSION_GENERATION` 并落盘到 `audit/errors.jsonl`。

## 7. C同学详细任务清单（Medgemma 1.5部署 + 按模态单独分析 + 自动化报告生成）

### C-1 任务：Medgemma 1.5 本地部署稳定化与调用接口

做什么：把 Medgemma 1.5 变成一个“稳定可调用”的后端组件，支持超时、失败重试与可插拔替换。

怎么做（步骤）：

1) 确认部署形态：优先选择“本机 HTTP 推理服务”或“Python进程内调用”二选一，并固定对外接口形式，避免团队成员各自调用方式不一致。

2) 资源策略：明确 GPU/CPU 优先级与最大并发数，设置单次推理超时，超时返回可重试错误。

3) 编写适配层接口：定义 `infer(prompt_id, payload) -> raw_text` 并在上层统一做 parse/validate/repair，确保未来替换为自训模型时，上层编排与前端展示协议不改动。

4) 记录审计：保存原始输出到 `static/processed/<file_id>/audit/medgemma_raw_<module>.json`，便于复盘与调参。

验收标准：

1) 在无影像输入的“空payload”或最小payload下仍能稳定返回；

2) 连续调用多次不会崩溃；

3) 超时/失败有明确错误码并可被上层降级处理。

### C-2 任务：按模态专属提示词与输出边界（按本文范围内定义执行）

做什么：为 NCCT/CTA(mCTA)/CTP 各自设计专属 prompt 与专属输出边界，每模态单独分析并输出各自结构化 JSON 与模态段落。

怎么做（步骤）：

1) 确认模态边界（以本文第2节第3条为准）：NCCT 偏出血排除/早期缺血/ASPECTS，CTA/mCTA 偏 LVO 部位与侧支提示，CTP 偏核心/半暗带/不匹配与 DEFUSE3 或 DAWN 风格量化提示，关键量化数值必须来自卒中量化模块。

2) 设计 prompt 版本化：以 `medgemma.ncct.v1`、`medgemma.cta.v1`、`medgemma.ctp.v1` 命名，并在模块结果 JSON 的 `model.prompt_version` 记录。

3) 设计输入组织 Evidence Pack：不直接喂 NIfTI，全都通过“关键切片PNG + QC摘要 + B输出的量化指标 + 缺失模态信息”组织为 payload。

4) 设计输出为严格 JSON：至少包含 findings 列表（conclusion/confidence/uncertainty/recommendations/structured），由上层统一 envelope 包装。

5) 输出边界约束：

   5.1) NCCT 不允许输出闭塞部位与侧支评分；

   5.2) CTA/mCTA 不允许凭空输出核心/半暗带体积；

   5.3) CTP 的核心/半暗带/不匹配数值必须引用 B 的 `stroke_v1.json`，模型只能做语言化解释与结构化归因。

验收标准：四组合逐一运行时，存在的模态都会生成对应的 `ncct_v1.json/cta_v1.json/ctp_v1.json` 与 `report_fragment_*.md`，缺失模态会生成固定缺失段落而不是空白。

### C-3 任务：多模态汇总报告自动生成（融合B的卒中量化证据）

做什么：把各模态段落与卒中分析数据融合，生成符合医学书写习惯的总报告草稿，强调结论与证据，并能在模态间冲突时保守表述。

怎么做（步骤）：

1) 获取“本次病例有哪些模态、哪些缺失”：从 `case_manifest.json` 的 `available_modalities` 读取，或通过 `GET /api/agent/result` 的索引读取。

2) 报告模板设计为分段 Markdown：按“检查方法、NCCT、CTA/mCTA、灌注与量化、总结与建议、证据索引”的顺序拼装。

3) 注入 B 的量化数据：把 `stroke_v1.json` 的 `core_volume_ml/penumbra_volume_ml/mismatch_ratio/threshold_basis/overlays` 写入 CTP 段落，必须标注单位、阈值依据与证据图路径。

4) 冲突表述策略：当 NCCT 提示大面积缺血而 core_volume 接近0 或 CTA侧别与灌注侧别冲突时，输出 indeterminate 并强制加“人工复核”建议，冲突检测可先做最小规则集合，详细融合推后。

5) 降级模板：当 Medgemma 不可用或输出不可解析时，使用固定模板生成每模态段落并在报告头部标注“模型输出不可用，已降级”。

输出格式：必须落盘 `static/processed/<file_id>/report/report_draft_v1.md`，并可选输出 `report_structured_v1.json` 供前端渲染。

验收标准：四组合逐一生成报告草稿；报告中包含指定字段（ASPECTS或indeterminate、闭塞部位或indeterminate、核心/半暗带/不匹配指标或缺失说明），并标注来源模态；模型不可用时仍能生成可读报告且错误码可见。

## 8. D同学详细任务清单（RAG对话：patient_id直读报告并脱敏后问答）

### D-1 任务：patient_id 到报告的检索路径与索引策略

做什么：医生输入 `patient_id` 后系统能定位到该病人的最新报告内容与结构化关键字段，无需医生再上传信息。

怎么做（步骤）：

1) 明确 patient_id 与 file_id 关系：优先从 Supabase 读取病人记录并获取最新关联 file_id（读取逻辑参考 [`get_patient_by_id()`](app.py:73) 与 [`api_get_patient()`](app.py:1251)）；若 Supabase 无法提供关联，则从本地索引文件查找（例如在 `case_manifest.json` 内写入 patient_id 并建立 `patient_id -> file_id` 的索引文件）。

2) 读取报告优先级：优先读取 `report_final_v1.md`，不存在则读取 `report_draft_v1.md`；同时读取 `module_results/*.json` 作为结构化事实来源。

3) 缺失与错误提示：病人不存在返回 `ERR_CASE_NOT_FOUND`；报告未生成返回“正在处理”并给出 `GET /api/agent/status` 的轮询链接；模态缺失则在回答中显式说明依据不足。

### D-2 任务：脱敏规则清单与实现方式

做什么：在进入检索与回答前完成去隐私化，至少脱敏姓名。

怎么做（步骤）：

1) 脱敏规则：姓名替换为“患者”或“Pxxxx”；手机号、身份证、地址用正则打码；时间只保留相对时间；日志与审计文件不得写明文姓名。

2) 实现方式：对报告 Markdown 与结构化 JSON 做一次 sanitization，生成 `rag_context_sanitized.json` 或内存对象；保留原文仅在受控目录并记录审计。

3) 失败兜底：脱敏失败则拒绝进入对话并提示“数据格式异常需人工处理”，避免泄露。

### D-3 任务：RAG检索粒度、引用策略与防幻觉约束

做什么：回答必须尽量引用原报告证据与结构化字段，不得编造。

怎么做（步骤）：

1) 检索粒度：先以“报告按段落”切分（NCCT段/CTA段/CTP段/总结段），必要时结合结构化 JSON 的 findings 做精确检索。

2) 引用策略：回答必须返回 `sources[]`，至少包含引用的报告段落标题与文件路径；当问题超出报告范围，必须回答“不确定/报告未提供”。

3) 权限边界：最小实现为“登录态或简单token”，并在审计日志记录 operator、patient_id、trace_id。

验收标准：给定存在的 patient_id 能稳定回答；回答文本中不出现明文姓名；sources 可追溯到 `report_*` 文件；病人不存在、报告未生成时提示明确且可继续路径清晰。

## 9. 集成联调策略（有先后顺序但无具体时间表）

集成联调按依赖链推进，保证每一步都可回退到旧流程。

第一阶段先完成“上传放宽 + manifest落盘 + viewer不崩”，即 A-1 与 B 的 manifest/QC字段联调，确保四组合至少能正确识别并进入 viewer。

第二阶段完成“卒中分析自动触发 + 证据可展示”，即 B 把 `stroke_v1.json` 与叠加图落盘稳定，A 的 viewer 能渲染 stroke 卡。

第三阶段完成“Medgemma按模态独立输出 + 报告草稿自动生成”，即 C 的三模态结果文件与 report_draft 能落盘，A 的 viewer 能预览 report。

第四阶段完成“RAG patient_id直读与脱敏问答”，即 D 能读到最新报告并脱敏回答，且不影响原有对话入口。

每次联调必须做旧流程回归：上传mCTA四件套、viewer切片浏览、对比度调节、伪彩生成、手动卒中分析按钮、报告保存。

## 10. 测试与验收（四组合 + 回归点）

四组合验收用例（每个用例必须可复现并能截图/录屏）：

1) NCCT-only：上传成功→自动编排→生成 ncct_v1.json 与 report_draft→viewer 动态展示 NCCT 卡与缺失CTA/CTP提示。

4) NCCT + CTA + CTP 参数图：上传成功→自动触发卒中分析→生成 stroke_v1.json 与 ctp_v1.json→viewer 展示量化与叠加图→报告包含核心/半暗带/不匹配指标与阈值依据。

关键回归点（必须每次联调都跑）：对比度调节按钮与拖拽调整仍可用（初始化见 [`initializeContrastController()`](static/js/viewer.js:104)）；伪彩生成按钮仍可用（触发见 [`togglePseudocolor()`](static/js/viewer.js:240)）；手动卒中分析按钮仍可用（模板按钮见 [`templates/patient/upload/viewer/index.html`](templates/patient/upload/viewer/index.html) 中的“脑卒中分析”，后端路由见 [`analyze_stroke()`](app.py:955)）。

## 11. 风险清单与应对/回滚

风险一是新功能破坏旧入口，回滚策略是所有新增能力必须通过新增 API 与新增产物目录实现，旧路由不改语义；上传改造只做“放宽校验+增加字段”，并保留 legacy 开关；viewer 只新增结果栏，出现问题可隐藏结果栏回退。

风险二是 Medgemma 输出不可解析或服务不稳定，应对是 C 必须实现 parse+validate+repair 重试与降级模板，且把原始输出落盘；报告生成不得因为模型失败而中断。

风险三是外部CTP与生成CTP在维度/切片数/方向上不一致，应对是 B 输出 QC warnings 并在缺陷情况下降级为仅展示可用指标，必要时跳过量化并提示补采。

风险四是性能与超时，应对是编排器为每个模块设置超时与部分完成策略，确保 viewer 可用且用户能看到“正在处理/失败原因”。

风险五是隐私与脱敏，应对是 D 的对话入口必须先脱敏再回答，日志不得落明文姓名；若无法保证脱敏则拒绝回答并提示。
