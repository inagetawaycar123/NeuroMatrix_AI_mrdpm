# 卒中智能体寒假执行计划（5人新手团队，可直接照着开发落地，优先P0且不破坏现有系统）

本文严格对齐寒假计划要点（见 [`plans/卒中智能体寒假计划.docx`](plans/卒中智能体寒假计划.docx)）以及既有设计与需求优先级（[`docs/CORE_FUNCTIONS.md`](docs/CORE_FUNCTIONS.md)、[`plans/STROKE_AGENT_PROPOSAL.md`](plans/STROKE_AGENT_PROPOSAL.md)、[`plans/STROKE_AGENT_RND_REQUIREMENTS.md`](plans/STROKE_AGENT_RND_REQUIREMENTS.md)、[`plans/STROKE_AGENT_ARCHITECTURE_EXECUTABLE.md`](plans/STROKE_AGENT_ARCHITECTURE_EXECUTABLE.md)、[`plans/STROKE_AGENT_FINAL_BLUEPRINT.md`](plans/STROKE_AGENT_FINAL_BLUEPRINT.md)）。本计划以“现有Flask后端+现有viewer/report页+既有MRDPM灌注生成与卒中分析能力”为基线，承诺不破坏现有流程入口（上传、viewer、对比度调节、伪彩生成、手动卒中分析、报告保存）。

我作为负责人会按“先冻结接口契约再并行开发、每周可演示、任何失败可解释且可回退”的方式推进。

## 一、寒假MVP范围与不做清单（按优先级交付，先P0再P1）

寒假交付目标分成两层。第一层是P0最小闭环，必须在现有Web端实现四种模态组合的自动识别与自动编排入口，并至少保证三种组合可稳定演示且不崩，同时不影响现有mCTA四件套流程。第二层是P1能力补齐，完成 CTA或mCTA 单独分析与多模态汇总报告的正式形态，并把前端动态展示做完整。

P0必须交付的功能边界如下。第一，上传支持更贴近临床的模态组合，NCCT为必选，其余可选，组合覆盖 NCCT-only、NCCT+single-phase CTA、NCCT+mCTA三期、NCCT+CTA+CTP参数图输入，并在上传成功后返回 `available_modalities` 与 `qc_summary`；此改动点必须兼容原有 [`upload_files()`](app.py:2803) 的返回结构，并只做“增加字段，不删字段”。第二，新增自动编排接口三件套 `POST /api/agent/run`、`GET /api/agent/status`、`GET /api/agent/result`，上传成功后自动触发一次编排任务，不需要医生在viewer里再点按钮；这三接口以 `file_id` 为主键，幂等键为 `file_id + idempotency_key`。第三，无论CTP参数图来自医生上传还是系统生成，只要能获得灌注参数图，就自动触发并复用现有卒中分析逻辑（半暗带/核心/不匹配）与可视化叠加图，复用入口以 [`analyze_stroke_case()`](stroke_analysis.py:374) 的能力为准，同时保持现有 [`analyze_stroke()`](app.py:955) 手动按钮仍可用。第四，前端viewer页面根据产物动态展示模态，不改变原有对比度调节与伪彩入口，新增一个“智能体结果栏”显示结构化结论卡片、证据图链接、处理状态机、缺失模态解释与下一步建议。

P0明确不做的内容如下。Medgemma 1.5 的“读片质量是否准确”不作为验收指标，验收只看输出结构化可解析、证据可追溯、失败可解释；CTA或mCTA 的侧支评分与LVO部位提示允许先做占位结论，正式规则与一致性校验推到P1；融合冲突卡与跨模态一致性校验推到P1；RAG对话的 patient_id 直读能力推到P1后半段或寒假末尾作为加分项，P0只要求预留接口与数据读取打通，不要求向量检索完整上线。

## 二、按周里程碑计划（每周都有可演示产物，按依赖顺序推进）

寒假以5周推进最稳妥，原因是新手团队需要留出充足联调与回归时间，同时必须保证每周都有可演示结果，避免最后集中爆炸。

第1周交付“主干可跑通”。演示内容是NCCT-only上传后自动开始处理，viewer里出现处理状态条与一张NCCT关键切片证据图，`GET /api/agent/status` 能从 running 变成 done，落盘生成 `static/processed/<file_id>/case_manifest.json` 与 `static/processed/<file_id>/module_results/ncct_v1.json` 两个文件，且不影响原有mCTA四件套上传与viewer浏览。

第2周交付“CTP参数图输入路径+卒中分析自动触发”。演示内容是NCCT+CTA+CTP参数图输入后自动生成卒中量化指标与叠加图，viewer里出现核心/半暗带/不匹配卡片与叠加图链接，伪彩生成入口仍可用，落盘生成 `static/processed/<file_id>/module_results/stroke_v1.json` 与 `static/processed/<file_id>/report/report_draft_v1.md`。

第3周交付“NCCT+mCTA三期→MRDPM生成灌注图→卒中分析自动触发”。演示内容是沿用现有mCTA四件套上传仍可跑，并在满足条件时自动触发灌注生成与卒中量化，落盘生成 perfusion 相关产物并在manifest标记 `perfusion_generated=true`，失败则以错误码提示但viewer仍可用。

第4周交付“CTA或mCTA单独分析模块+按模态分段的总报告拼装”。演示内容是对存在CTA或mCTA的病例，自动生成 `cta_v1.json` 与 `report_fragment_cta.md`，并由汇总器拼装包含NCCT段、CTA段、CTP段的总报告草稿；此处允许CTA结论先由Medgemma产生结构化摘要，侧支与闭塞部位先以保守措辞输出并带不确定性。

第5周交付“收敛与验收+RAG patient_id 直读最小实现”。演示内容是四种组合全覆盖，缺失模态解释文案一致且可追溯，失败与部分完成仍可用；对话页支持输入 patient_id 自动读取该病例报告并脱敏后回答，若病例不存在或报告未生成有可执行提示路径；完成导师验收脚本与回滚预案。

## 四、每个功能的可执行实现步骤（从读现有文件到新增路由与落盘）

本节把每个人负责的功能拆成“可以直接开工”的步骤，格式固定为先读哪些现有文件与入口函数，再改哪些位置，再新增哪些文件/函数/路由，再怎么联调验证。技术栈选择以“最小侵入、与现有依赖兼容、可插拔替换”为原则。

### A的实现步骤：编排器API与报告拼装器

第一步阅读现有后端入口与关键路由，确认不会改坏现有功能，阅读 [`upload_files()`](app.py:2803)、[`viewer_page()`](app.py:2798)、[`generate_all_pseudocolors_route()`](app.py:895)、[`analyze_stroke()`](app.py:955)、[`api_save_report()`](app.py:1276)。第二步在 [`app.py`](app.py:1) 新增三条路由 `POST /api/agent/run`、`GET /api/agent/status`、`GET /api/agent/result`，并在文件内新增一个“编排主函数”作为纯Python函数供路由调用，要求该函数只依赖 `file_id` 与manifest路径，不依赖Flask request对象，便于单测。第三步实现幂等逻辑，幂等键采用 `file_id + idempotency_key`，若 `audit/job_v1.json` 已存在且记录相同幂等键，则直接返回已有job_id与result索引。第四步实现 stage 与 modules 状态字典，最小包含 ncct、cta、ctp、stroke、report 五项，P0允许cta为skipped。第五步实现报告拼装器，输入为 `module_results/*.json` 与 `report_fragment_*.md`，输出为 `report/report_draft_v1.md`，报告必须按模态分段并引用证据路径，模板对齐 [`plans/STROKE_AGENT_FINAL_BLUEPRINT.md`](plans/STROKE_AGENT_FINAL_BLUEPRINT.md)。第六步实现审计落盘与错误码，任何异常写入 `audit/errors.jsonl` 并在 status 返回 `ERR_*`。

技术栈建议为原生Python标准库加现有依赖，JSON用内置 `json`，日志用内置 `logging` 写文件，路径用 `os.path`；不引入新框架以降低新手维护成本。未来替换为异步队列时，仅需要把编排主函数搬到后台任务执行，接口字段不变。

### B的实现步骤：上传改造、模态识别、manifest落盘、QC

第一步阅读上传页与上传脚本，定位现有强制四件套校验，阅读 [`templates/patient/upload/index.html`](templates/patient/upload/index.html) 与 [`processFiles()`](static/js/upload.js:72)。第二步修改前端校验逻辑，使按钮可在“已选择NCCT”时启用，其他文件可为空，同时在前端将已选择的文件角色清晰传给后端，例如 `formData.append('ncct_file', ...)` 必须存在，其它 append 按存在性追加。第三步阅读后端上传入口 [`upload_files()`](app.py:2803)，改造为允许缺失 vcta/dcta 或外部 ctp_maps，并在响应体新增字段 `available_modalities` 与 `qc_summary`，保持原有字段仍返回以兼容viewer现有逻辑。第四步实现manifest落盘，路径为 `static/processed/<file_id>/case_manifest.json`，内容字段至少包含 `patient_id/file_id/available_modalities/qc/modules`，并创建目录结构 `inputs/module_results/report/audit`。第五步实现最小QC，P0只做可读性与体数据维度检查，输出 `qc.level` 与 `warnings[]`，读不到则 `qc.level=fail` 并返回错误码 `ERR_QC_FAIL`。

技术栈建议继续沿用现有Flask上传处理方式，不引入额外对象存储。若需要读取NIfTI头信息，优先使用当前项目已有方式；若缺少库，可引入 `nibabel` 作为唯一新增依赖，但必须先确认与现有 `requirements.txt` 兼容并在Windows下可安装。

### C的实现步骤：CTP来源统一的卒中量化自动触发与证据落盘

第一步阅读现有卒中分析模块与其文件组织方式，阅读 [`class StrokeAnalysis`](stroke_analysis.py:16) 与 [`analyze_stroke_case()`](stroke_analysis.py:374)，确认它当前从 `static/processed/<file_id>/` 下查找 `slice_*_tmax_output.npy` 并生成 `stroke_analysis/*.png`。第二步定义P0下的统一输入适配规则，使外部CTP参数图输入能落到与当前分析函数一致的文件命名方式，最少要能生成若干 `slice_XXX_tmax_output.npy` 与 `slice_XXX_mask.npy`，或者在编排器里新增一层适配函数，读取外部参数图并按切片导出为该命名格式。第三步在编排器执行流程中加入“自动触发卒中分析”步骤，触发条件与优先级严格对齐 [`plans/STROKE_AGENT_FINAL_BLUEPRINT.md`](plans/STROKE_AGENT_FINAL_BLUEPRINT.md) 的规则，即外部CTP存在则直接分析，否则mCTA且允许生成则先生成灌注再分析。第四步把卒中分析结果转换为 `module_results/stroke_v1.json` 的结构化输出，至少包含核心/半暗带/不匹配三指标、阈值依据字符串、叠加图路径数组，并保证叠加图可通过现有 [`get_stroke_analysis_image()`](app.py:1006) 访问。第五步在失败时写入 `ERR_STROKE_ANALYSIS`，并生成降级报告段落说明缺失原因，不阻塞NCCT段落生成。

技术栈建议复用现有 `numpy`、`opencv-python`、`matplotlib`、`scipy` 依赖，不新增深度学习模型。外部CTP参数图若是NIfTI优先用 `nibabel` 读取，再用 `numpy` 切片落盘。

### D的实现步骤：viewer动态布局、状态机、结果卡片、报告预览

第一步阅读现有viewer结构与脚本初始化流程，阅读 [`templates/patient/upload/viewer/index.html`](templates/patient/upload/viewer/index.html) 与 [`initializeViewer()`](static/js/viewer.js:58)，确认现有图像格子与对比度控制器不改动。第二步在viewer模板中新增一个容器区域作为“智能体结果栏”，并新增一个状态条区域，用于展示 uploading/processing/done/failed/partial 状态。第三步在 [`static/js/viewer.js`](static/js/viewer.js) 中新增两个函数，一个用于轮询 `GET /api/agent/status` 并更新状态条，一个用于拉取 `GET /api/agent/result` 并根据 `case_manifest.available_modalities` 渲染卡片。第四步卡片渲染规则固定化，NCCT卡显示出血排除、早期缺血、ASPECTS字段（如果缺失就显示indeterminate），stroke卡显示核心/半暗带/不匹配与叠加图链接，CTA卡在P0可显示“待上线”或skipped原因但不得空白。第五步缺失模态文案必须从manifest与模块状态生成，文案模板写死但触发条件不能靠猜测。第六步新增“查看预报告”按钮，打开 `report/report_draft_v1.md` 的内容预览，P0可以用简单弹窗或侧边栏渲染，不要求富文本编辑。

技术栈建议保持原生JS与现有CSS体系，不引入前端框架，以降低新手复杂度。渲染可用现有字符串拼接与DOM API，网络请求统一用 fetch，错误提示复用现有 showMsg 或 showMessage 方法。

### E的实现步骤：测试矩阵、数据包、验收脚本、RAG最小实现与脱敏检查

第一步建立测试矩阵与每周Demo脚本，四组合分别写出“输入文件角色、预期available_modalities、预期生成文件、预期viewer显示卡片”。第二步建立回归清单，必须包含“原有mCTA四件套上传仍可用、对比度调节仍可用、伪彩生成仍可用、手动卒中分析按钮仍可用、报告保存仍可用”。第三步从第2周开始，每周Demo前按脚本走通并录屏保存，失败必须记录 `ERR_*`、截图、复现步骤，提交到issue。第四步在第5周实现RAG最小闭环，新增对话入口允许输入 patient_id，后端读取该patient_id对应的最新报告文本，执行最小脱敏规则（姓名替换），再把报告作为上下文回答问题；若病例不存在或报告未生成必须返回明确提示并给出下一步路径。第五步对审计与日志做检查，确认 `audit/job_v1.json` 不写明文姓名。

技术栈建议测试以“脚本+人工验收”为主，必要时可写一个最小Python脚本调用接口做回归（不要求pytest全套）。RAG最小实现不引入向量库，先用“全文上下文+引用来源路径”的方式满足功能要求，后续再扩展向量检索。

## 五、跨人协作方式与接口契约（必须先冻结后开发）

协作的第一条纪律是接口契约先行。第1周第3天我会组织一次契约冻结会，冻结三件事：`file_id` 作为病例目录主键与其传递方式；`case_manifest.json` 的字段集合与文件路径；模块结果JSON envelope v1 的必填字段与错误码集合。冻结后的契约由A维护，任何变更必须在PR里同步更新 [`plans/STROKE_AGENT_FINAL_BLUEPRINT.md`](plans/STROKE_AGENT_FINAL_BLUEPRINT.md) 或本文件并经过review。

接口契约的关键点如下。`file_id` 必须从上传响应返回并写入浏览器端状态（现有viewer已在 [`initializeViewer()`](static/js/viewer.js:58) 使用 `currentFileId`），编排器只接受 `file_id` 不接受文件路径。产物目录根必须固定为 `static/processed/<file_id>/`，目录下必须包含 `case_manifest.json`、`module_results/`、`report/`、`audit/`。模块结果JSON必须包含 `schema_version/module/file_id/patient_id/job_id/qc/evidence/findings/model/status/audit`，任何失败必须有 `status.error.code`。前端展示协议是viewer只调用 `/api/agent/status` 与 `/api/agent/result`，并且只按返回字段渲染，不允许前端自行推断“应该有什么模态”。Git协作采用 main + feature 分支，PR至少两人review，涉及 [`app.py`](app.py:1) 的变更必须A review，涉及viewer页面必须D review。

联调流程固定为每周两次：周三晚做后端接口联调，验证返回体字段与落盘文件存在性；周六晚做端到端联调，从上传到viewer到报告预览走一遍。每日同步节奏为10分钟站会，只汇报阻塞与接口变化。

## 六、风险点与回退方案（保证不破坏现有系统）

寒假交付最大的风险是“新功能影响旧功能”。回退方案统一为“新增而不改坏”：新增 `/api/agent/*` 与新增目录，不删除任何旧路由；上传改造只做“放宽校验与增加字段”，若出问题可以临时保留原四件套入口并加一个“兼容模式开关”；viewer改造只新增结果栏，不动现有图像格子与对比度控制器。性能与超时风险通过在编排器里加入模块级超时与部分完成策略缓解，即某模块失败不阻塞其余模块与viewer浏览。

模型效果不稳定风险通过“输出允许indeterminate但必须可解析并有证据”缓解。NIfTI兼容性风险通过P0明确只支持 `.nii/.nii.gz` 并在上传时校验缓解。导师验收口径风险通过E维护的每周Demo脚本与最终验收脚本缓解，脚本以可演示产物与可测字段为准，不以模型准确率为准。
